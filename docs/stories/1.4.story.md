# Story 1.4: Suíte Automatizada de Testes RLS

## Status

InReview

## Story

**Como** mantenedor da base KEYRA,
**Eu quero** ter uma suíte pgTAP que execute em CI validando o isolamento multi-tenant de todas as tabelas `org_id`-scoped,
**Para que** qualquer regressão em RLS (accidental `DISABLE ROW LEVEL SECURITY`, policy errada, JWT claim perdido) seja detectada antes do deploy — nunca em produção com dados de clientes.

## Contexto

ADR-011 coloca RLS como peça central de isolamento entre organizações. Hoje dependemos de confiança manual de que 21 tabelas têm `ENABLE RLS` + policies corretas. A migration `20260416001700_rls_policies.sql` é longa e facilmente quebrável em um refactor. Precisamos de um *safety net* automatizado que rode em toda PR.

**Decisão técnica — plain SQL em vez de pgTAP:** A Story 0.4 já entregou `supabase/tests/rls_isolation.test.sql` em plain SQL com asserções via `RAISE EXCEPTION`. Essa abordagem funciona para os cenários de RLS (onde a pergunta central é sempre "quantas linhas esta query retorna?") e não exige a extension `pgtap` extra (que precisaria ser habilitada tanto localmente quanto em CI). Manter plain SQL evita mistura de sintaxes, roda via `psql -v ON_ERROR_STOP=1` (ferramenta ubíqua) e é mais fácil de estender — basta mais um bloco `DO $$ ... $$`. O escopo desta story é **expandir cobertura + automatizar em CI**, não refatorar para pgTAP.

## Complexidade

**T-shirt size:** S–M (~4–6 pontos · 0,5–1 dia)

Escopo é definido e repetitivo: 21 tabelas · ~3 asserções por tabela · infra CI standard.

## Acceptance Criteria

### AC1 — Suíte plain SQL local (cobertura expandida)

1. Existe `supabase/tests/rls_isolation.test.sql` (já criado na Story 0.4) **expandido** para cobrir as seguintes tabelas tenant-scoped além das 3 existentes (customers, services, professionals): `memberships`, `organization_invites`, `appointments`, `commands`, `transactions`, `supplies`, `inventory_movements`, `payments`.
2. Para cada tabela coberta, valida no mínimo duas asserções:
   - `SELECT` como user A retorna somente registros de org A (count=1 com fixture mínima)
   - `UPDATE` / `DELETE` de registro de org B por user A retorna 0 linhas (testa a policy USING)
3. Um cenário adicional valida bloqueio de `INSERT` cross-tenant (`WITH CHECK` policy) em pelo menos 3 tabelas críticas (customers, services, memberships).
4. Existe bloco de **smoke inverso** que, se alguém desabilitar `ENABLE ROW LEVEL SECURITY` em qualquer tabela coberta, a suíte falha com mensagem explícita (proteção contra regressão — confere via `pg_class.relrowsecurity`).
5. Rodando `psql -v ON_ERROR_STOP=1 -f supabase/tests/rls_isolation.test.sql` contra um Postgres com as migrations aplicadas, o script termina com `NOTICE: ALL RLS ISOLATION TESTS PASSED` e exit code 0.

### AC2 — GitHub Actions CI

1. Existe `.github/workflows/rls-tests.yml` disparado em `push` para `main` e em `pull_request` visando `main` (ignorando changes em `docs/**` e `*.md` para não rodar em doc-only PRs).
2. O job usa o runner `ubuntu-latest` e um container Postgres 17 (via service container do GH Actions, mais rápido que subir a stack inteira do `supabase start`).
3. O job aplica todas as migrations de `supabase/migrations/*.sql` em ordem alfabética via `psql` contra o Postgres do service container.
4. O job roda a suíte com `psql -v ON_ERROR_STOP=1 -f supabase/tests/rls_isolation.test.sql` — qualquer `RAISE EXCEPTION` faz o step falhar (exit code ≠ 0).
5. Tempo total do workflow ≤ 3 minutos em execução média (Postgres container inicia em ~10 s).

### AC3 — Badge e documentação

1. O `README.md` principal ganha badge de status do workflow `RLS Tests` (via `github.com/{owner}/{repo}/actions/workflows/rls-tests.yml/badge.svg`).
2. Existe `docs/testing/rls-tests.md` (novo, curto — ≤ 150 linhas) explicando: o que os testes cobrem, como rodar localmente (`supabase start && supabase test db`), como adicionar testes para uma nova tabela, como interpretar falhas comuns.

### AC4 — Hygiene

1. A suíte não pressupõe dados pré-existentes: cria + limpa tudo em `BEGIN; ... ROLLBACK;` (ou em schema próprio `tests` criado e dropado).
2. Nenhum teste depende de `NOW()` ou estado randômico não-determinístico — se houver, é controlado via `SET LOCAL TIME ZONE` ou seeds fixos.
3. Nenhuma credencial real (service_role_key etc.) é commitada; CI usa apenas o superuser local do `supabase start`.

## Dev Notes

- **pgTAP simula JWT** via `request.jwt.claims` configurado com `SET LOCAL`. Nosso hook `custom_access_token_hook` injeta `org_id`, então os testes precisam setar o claim manualmente antes de cada cenário.
- **Helper para "rodar como user X":** função `tests.jwt_as(user_id, org_id)` seta os três GUCs (`request.jwt.claim.sub`, `role`, `request.jwt.claim.org_id`). Isso replica o efeito de autenticação sem tocar em `auth.users`.
- **Cenário mínimo:** 2 orgs, 2 users, 2 memberships. Cada tabela recebe 1 registro em cada org. A asserção "user A vê só org A" é `SELECT count(*) FROM tabela WHERE org_id = $1 → 1` e `SELECT count(*) FROM tabela WHERE org_id = $2 → 0`.
- **INSERT bloqueado** usa `throws_ok()` do pgTAP: envolvendo o INSERT em anon block e verificando que lança `new row violates row-level security policy`.
- **Tempo de execução:** cada teste é rápido (<50 ms) mas `supabase start` demora ~40 s. Em CI sem cache, o workflow roda em ~2–3 min. Dentro do aceitável.
