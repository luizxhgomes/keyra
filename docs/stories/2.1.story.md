# Story 2.1: CRUD de Pacientes

## Status

InReview

## Story

**Como** owner/admin/professional de uma organização KEYRA,
**Eu quero** cadastrar, editar e buscar pacientes,
**Para que** eu possa montar a base de CRM que alimenta agenda, comanda e DRE.

## Complexidade

**T-shirt:** M (~6 pontos · 1 dia)

## Contexto

`customers` já existe (migration 006) com campos nome, telefone, email, nascimento, notas e placeholders para CPF criptografado (cpf_encrypted bytea + cpf_hash text — ADR-017, LGPD). Para o MVP, **CPF é fora do escopo imediato** (pode ser ativado em Phase 5 quando a função `pgp_sym_encrypt` + `COLUMN_ENCRYPTION_KEY` estiverem testadas end-to-end). Story 2.1 entrega apenas os campos em texto claro + busca fuzzy por nome.

## Acceptance Criteria

### AC1 — Listagem

1. Página `/pacientes` protegida por `requireAuth()` + role `viewer`+.
2. Lista paginada (20 por página) com colunas: nome, telefone, email, próxima visita (placeholder até Story 2.4), ações.
3. Busca por nome usando o índice `customers_full_name_trgm` (`ilike %q%` no server, descarregado pela gin_trgm).
4. Estado vazio: "Nenhum paciente cadastrado" + botão "Novo paciente".

### AC2 — Criar

1. Botão "Novo paciente" abre form em drawer/sheet (shadcn) com: nome* (1–200), telefone (E.164 parcial, máscara BR), email, data de nascimento (date picker), notas.
2. Validação Zod no server; erros inline no form.
3. Após salvar: toast "Paciente cadastrado" + redireciona para lista atualizada.

### AC3 — Editar

1. Clicar no nome abre `/pacientes/[id]` ou o mesmo drawer em modo edit.
2. Todos os campos mutáveis. `created_at` e `id` são read-only.
3. `updated_at` é mantido pelo trigger.

### AC4 — Desativar

1. Ação "Arquivar" faz soft-delete (`deleted_at = now()`).
2. Pacientes arquivados não aparecem na lista por default. Toggle "mostrar arquivados" opcional (fora do MVP da story).
3. Aviso: "Registros históricos (agenda, comanda) permanecem intactos."

### AC5 — Segurança e multi-tenant

1. Todas as Server Actions validam `requireAuth()` e RLS; `org_id` nunca vem do client.
2. Não é possível ver pacientes de outra org (RLS cobre via Story 1.4).
3. Papel `viewer` só lê; `professional` pode criar/editar paciente.
