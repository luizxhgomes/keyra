# Story 2.3: Insumos e Rateio por Serviço

## Status

Done

## Story

**Como** owner/admin,
**Eu quero** cadastrar insumos (supplies) e vincular quantidades por serviço (BOM),
**Para que** o rateio de estoque seja automático quando o atendimento for concluído (trigger `_consume_command_inventory` já existe) e o custo unitário do serviço seja rolado do BOM.

## Complexidade

**T-shirt:** M (~6 pontos · 1 dia)

## Contexto

Schemas prontos: `supplies`, `service_supplies` (M:N com quantity) e `inventory_movements` (append-only). Trigger `_consume_command_inventory` já cria movement (-) quando command_item é inserido.

## Acceptance Criteria

### AC1 — Insumos (CRUD)

1. `/estoque/insumos` (ou `/estoque`) protegido por owner/admin.
2. Lista com: nome, unidade, custo unitário, estoque atual (cache), nível de recompra, fornecedor, ativo.
3. Criar/editar: nome*, unit (default "un"), unit_cost, reorder_level opcional, supplier_name opcional.
4. Soft-delete via `deleted_at`.

### AC2 — Ligação com serviço (BOM)

1. Na edição de um serviço (Story 2.2) existe tab "Insumos".
2. Adicionar insumo ao serviço escolhendo o supply e informando `quantity` (numeric).
3. Remover insumo do BOM.
4. Ao salvar, recalcular `services.unit_cost` como `SUM(service_supplies.quantity * supplies.unit_cost)` (via trigger/RPC ou recalculação na Server Action — decisão: recalcular no server para simplicidade).

### AC3 — Movimentações (somente leitura neste MVP)

1. `/estoque/movimentacoes` lista `inventory_movements` com colunas: data, supply, quantidade (+/-), motivo (`created_via` se existir; senão: "manual" vs "automático").
2. Sem UI de entrada manual agora — fica para Phase 5.

### AC4 — Segurança e multi-tenant

1. Todas as tabelas já têm RLS (Story 0.4); actions com `requireAuth()`.
2. Apenas owner/admin mutam.

## Dev Notes

### Decisões de implementação

1. **Recálculo do `services.unit_cost` na aplicação, não em trigger.** O AC2.4 deixa a decisão aberta; escolhi server-side por dois motivos: (a) deixa o cálculo visível e debuggável no código TS; (b) `attachSupplyToService` e `detachSupplyFromService` já fazem 1 round-trip mutativo, então adicionar 1 SELECT + 1 UPDATE é desprezível. Quando precisarmos de defesa em profundidade (ex.: import em massa que pula a aplicação), criamos um trigger SQL espelhando a fórmula. Não bloqueia esta entrega.
2. **Propagação reversa quando `supplies.unit_cost` muda.** Editar um insumo recompõe automaticamente todos os serviços que dependem dele (via `recalcServicesUsingSupply`). Sem esse passo, o BOM ficaria stale silenciosamente. Falha do recálculo NÃO bloqueia o upsert (logging silencioso) — preferi consistência eventual a um upsert que reverte mid-way.
3. **`attachSupplyToService` usa `upsert(onConflict)`** em vez de INSERT. A UNIQUE constraint `(service_id, supply_id)` torna o INSERT direto suscetível a 23505 quando o usuário re-adiciona um insumo ainda visível na lista (edge case raro, mas existe). O upsert resolve sem complicar UI.
4. **Picker filtra insumos já vinculados** (`availableForPicker`). UX limpa: o usuário não consegue escolher um item que daria erro pela UNIQUE. Quando todos foram vinculados, o select fica disabled com mensagem explícita.
5. **BOM só aparece para `type='service'`**. Produtos (kits/revenda) não têm BOM no schema KEYRA — `service_supplies.service_id` aponta para `services` mas semanticamente só faz sentido para serviços executados.
6. **Layout com sub-nav espelha padrão de `/servicos`**. Mantém consistência visual e cognitiva: o usuário aprende uma vez e reusa. `EstoqueSubNav` é Client porque depende de `usePathname`.
7. **Movimentações sem entrada manual neste MVP** (AC3.2). Toda escrita em `inventory_movements` vem do trigger `_consume_command_inventory` quando comanda é paga. Em Phase 5 adicionamos UI para entry/adjustment manuais.
8. **Sidebar não foi tocada** — já tinha item "Estoque" desde Story 1.1 com `href='/estoque'` apontando para a rota agora criada. Apenas reaproveitamento.
9. **RLS via security_invoker.** A migration 023 (aplicada agora-pouco) garante que qualquer SELECT/INSERT/UPDATE/DELETE em `supplies`/`service_supplies`/`inventory_movements` respeite a policy `org_id = current_org_id()`. As actions filtram explicitamente por `org_id` em todos os WHEREs como camada redundante (defesa em profundidade).

7. Remover o insumo do BOM (botão lixeira no editor) → toast "Insumo removido do BOM." e custo do serviço volta para 0.
8. `/estoque/movimentacoes` → vazio (sem comandas pagas ainda).
9. Tentar acessar `/estoque/insumos/novo` como usuário com role `viewer` ou `professional` → action `upsertSupply` rejeita com `AuthorizationError` (a página renderiza, mas o submit falha — UX poderia melhorar; não bloqueia esta story).
