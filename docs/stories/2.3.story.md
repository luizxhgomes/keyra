# Story 2.3: Insumos e Rateio por Serviço

## Status

Ready

## Story

**Como** owner/admin,
**Eu quero** cadastrar insumos (supplies) e vincular quantidades por serviço (BOM),
**Para que** o rateio de estoque seja automático quando o atendimento for concluído (trigger `_consume_command_inventory` já existe) e o custo unitário do serviço seja rolado do BOM.

## Complexidade

**T-shirt:** M (~6 pontos · 1 dia)

## Contexto

Schemas prontos: `supplies`, `service_supplies` (M:N com quantity) e `inventory_movements` (append-only). Trigger `_consume_command_inventory` já cria movement (-) quando command_item é inserido.

## Acceptance Criteria

### AC1 — Insumos (CRUD)

1. `/estoque/insumos` (ou `/estoque`) protegido por owner/admin.
2. Lista com: nome, unidade, custo unitário, estoque atual (cache), nível de recompra, fornecedor, ativo.
3. Criar/editar: nome*, unit (default "un"), unit_cost, reorder_level opcional, supplier_name opcional.
4. Soft-delete via `deleted_at`.

### AC2 — Ligação com serviço (BOM)

1. Na edição de um serviço (Story 2.2) existe tab "Insumos".
2. Adicionar insumo ao serviço escolhendo o supply e informando `quantity` (numeric).
3. Remover insumo do BOM.
4. Ao salvar, recalcular `services.unit_cost` como `SUM(service_supplies.quantity * supplies.unit_cost)` (via trigger/RPC ou recalculação na Server Action — decisão: recalcular no server para simplicidade).

### AC3 — Movimentações (somente leitura neste MVP)

1. `/estoque/movimentacoes` lista `inventory_movements` com colunas: data, supply, quantidade (+/-), motivo (`created_via` se existir; senão: "manual" vs "automático").
2. Sem UI de entrada manual agora — fica para Phase 5.

### AC4 — Segurança e multi-tenant

1. Todas as tabelas já têm RLS (Story 0.4); actions com `requireAuth()`.
2. Apenas owner/admin mutam.
