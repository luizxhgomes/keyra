# Story 2.5: Agendamento (criar appointment)

## Status

InReview

## Story

**Como** owner/admin/professional,
**Eu quero** agendar paciente + serviço + profissional em um horário,
**Para que** a agenda se torne fonte do faturamento e a receita prevista seja calculada automaticamente.

## Complexidade

**T-shirt:** M (~7 pontos · 1,5 dia)

## Acceptance Criteria

### AC1 — Form

1. Modal/drawer com: paciente (combobox busca fuzzy), serviço (combobox), profissional (select), data, horário início, duração (autofill de `services.duration_minutes` · pode ser sobrescrito).
2. Horário fim calculado automaticamente.
3. `price_snapshot` = `services.price` no momento do agendamento (armazenado imutável por ADR-013).
4. `commission_snapshot` = `services.commission_rate` ?? `professionals.default_commission_rate` ?? 0.

### AC2 — Validação

1. Server recusa se o EXCLUDE constraint falhar (double-book); mostra mensagem amigável "Horário indisponível para este profissional".
2. Recusa se `professional` não pertence à org ou está inativo.

### AC3 — UX

1. Integração com FullCalendar (Story 2.4): drag-and-drop e click em slot vazio abrem este form. **Story 2.4 entrega o evento de slot-click/drag exposto via callback; esta story (2.5) consome o callback e renderiza o form (drawer no mobile, modal no desktop).**
2. Após criar, agenda atualiza sem reload (via `revalidatePath('/agenda')`).
3. Erro de double-book (EXCLUDE constraint) → toast `sonner` com mensagem amigável: "Horário indisponível para este profissional. Escolha outro horário ou outro profissional." Não fecha o form — usuário pode ajustar e tentar de novo. Campo `notes` (textarea opcional, max 500 chars) presente no form mas tratado como livre.

## Dev Decisions

1. **Pickers via SSR (não fetch on-open)** — `page.tsx` carrega `listAgendaPickers()` em `Promise.all` junto com `listAgendaProfessionals()` e passa pickers como prop pro `CalendarClient` → `AgendamentoForm`. Motivação: evita `useEffect`+setState no client (regra `react-hooks/set-state-in-effect` do React 19) e simplifica o form. Trade-off: pickers ficam "stale" se o user adiciona paciente/serviço noutra aba — refresh manual da página resolve. Aceitável para MVP.
2. **price/commission convertidos para `number` no INSERT** — tipos gerados pelo Supabase mapeiam `numeric(14,2)` para `number`. Para os valores absolutos do KEYRA (até R$ 99,999,999,999.99) o range cabe sem perda em IEEE 754. O Postgres re-armazena como `numeric` preservando precisão. Decimal.js + `ROUND_HALF_EVEN` continua sendo o caminho de **agregação** (DRE), não de **transporte** ponto-a-ponto.
3. **Sentinela `__no_customer__`** no Select do paciente porque o radix-Select não aceita `value=""`. Mapeado pra `undefined` no submit e o banco aceita `customer_id NULL` (walk-in/sem identificação).
4. **`useWatch` em vez de `form.watch()`** — alinhado com regra `react-hooks/incompatible-library` (memoização do RHF não é segura com React Compiler).
5. **EXCLUDE constraint capturado pelo SQLSTATE** — Postgres retorna `23P01` em violação de exclusion. Mapeado pra mensagem amigável "Horário indisponível…" sem fechar o form (user pode ajustar).
