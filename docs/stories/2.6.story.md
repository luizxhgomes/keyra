# Story 2.6: Status de Agendamento

## Status

InReview

## Story

**Como** profissional,
**Eu quero** marcar um agendamento como realizado / cancelado / falta,
**Para que** a comanda seja gerada automaticamente no atendimento (trigger existente) e o DRE receba os dados corretos.

## Complexidade

**T-shirt:** S (~3 pontos · 0,5 dia)

## Acceptance Criteria

### AC1 — Transições

1. Do status `scheduled`: ir para `done`, `cancelled` ou `no_show`.
2. De `done`: apenas `cancelled` (caso de correção — estornar comanda é Phase 3+).
3. De `cancelled` / `no_show`: read-only (só permite editar notas).

### AC2 — UI

1. Sheet de detalhes (Story 2.4) mostra badge do status + botões contextuais.
2. Ao marcar `done`, trigger do banco gera `commands` e `command_items` (verificar que apareceu, mas Story 3.x polui a UI).
3. Ao `cancelled`, pedir motivo (armazenado em `cancel_reason`).

### AC3 — Auditoria

1. Trigger de audit_log registra todas as transições.

### AC4 — UX de cancelamento

1. Ao clicar "Cancelar" no sheet (Story 2.4), abre **AlertDialog** (radix/shadcn) com:
   - Combobox de motivos pré-definidos: "Paciente desmarcou", "Conflito de agenda", "Profissional indisponível", "Outro" (último libera campo livre).
   - Textarea opcional para detalhe (`cancel_reason` no banco grava a string final concatenada `motivo: detalhe`).
   - Botões "Não cancelar" (default focus) e "Confirmar cancelamento" (destructive variant).
2. Ao clicar "Falta (no_show)" — sem motivo, transição direta com confirmação simples.
3. Ao clicar "Concluir": transição direta + toast "Atendimento concluído. Comanda gerada automaticamente." (porque trigger `trg_appointments_done_to_command` cria a comanda).

## Dev Notes

### Decisões de implementação

1. **Transição validada em duas camadas (defesa em profundidade).** O Zod (`changeAppointmentStatusSchema`) restringe `to ∈ {done, cancelled, no_show}` e exige `cancelReason` quando `to=cancelled`. A action lê o `status` atual via Supabase (RLS isola por org) e impõe a matriz de transições da AC1: `scheduled → {done, cancelled, no_show}`, `done → cancelled`, demais bloqueados. Mensagem de erro carrega rótulos pt-BR ("Realizado", "Falta", etc.).
2. **`cancel_reason` formado client-side.** Combobox + textarea concatenam como `motivo: detalhe` quando há os dois; só motivo quando textarea vazio; só detalhe livre quando "Outro" (gating por `canSubmit`). O backend grava o trim direto.
3. **Trigger do banco continua sendo a fonte da comanda.** `changeAppointmentStatus` faz apenas o UPDATE do status; quem cria `commands` + `command_items` é `trg_appointment_done_creates_command()` (BEFORE UPDATE) na mesma transação. O `done_at` é setado pelo trigger via `COALESCE(NEW.done_at, now())`. Audit_log também é trigger (`trg_audit_appointments`), zero código de aplicação.
4. **Pattern "adjust state during render"** no `cancel-dialog.tsx` em vez de `useEffect`. ESLint 9 + React Compiler bloqueiam `setState` síncrono em effect; trocamos por comparação `prev/current` durante render — equivalente funcional, sem cascading renders. Ref: <https://react.dev/learn/you-might-not-need-an-effect#adjusting-some-state-when-a-prop-changes>.
5. **Botão Editar segue placeholder.** Story 2.5 entregou apenas criação. O botão fica `disabled` com label "Editar (em breve)" — uma futura story de edição reativa o botão. Decisão tomada porque editar exige form, validação de double-book em UPDATE e UI distinta de criação.
6. **Refetch após mutação.** Em vez de patchar evento em memória no FullCalendar, chamamos `refetchEvents()` via callback. Custo desprezível (cap de 200) e elimina drift entre cliente e banco — especialmente importante porque o trigger pode ter side effects que afetam visualizações futuras (ex.: filtros por status).
7. **Botão "Falta" só aparece em `scheduled`.** AC4.2 não menciona transição `done → no_show`, e a regra de negócio reflete isso — só faz sentido marcar como falta antes de o atendimento acontecer.

### Smoke test sugerido para QA

1. Abrir `/agenda`, clicar em evento `scheduled` → sheet mostra badge azul "Agendado", 3 botões visíveis.
2. Clicar **Concluir atendimento** → AlertDialog → confirmar → toast "Atendimento concluído. Comanda gerada automaticamente." → evento some/atualiza, sheet fecha, refetch executa.
3. Verificar no Supabase Studio: `SELECT * FROM commands WHERE appointment_id = '{id}'` deve retornar uma linha; `SELECT * FROM audit_log WHERE table_name = 'appointments' AND record_id = '{id}'` deve registrar a transição.
4. Clicar em outro evento `scheduled` → **Cancelar** → AlertDialog mostra combobox de 4 motivos + textarea. Submit desabilitado até escolher motivo. Selecionar "Outro" → submit fica desabilitado até digitar texto livre. Confirmar → toast "Agendamento cancelado." → evento aparece com `opacity-40` (já tratado em `calendar-client.tsx`).
5. Clicar evento `cancelled` → sheet mostra badge cinza "Cancelado" + aviso "somente leitura"; nenhum botão de transição visível.
6. Clicar em evento `done` → sheet mostra badge verde "Realizado" + botão "Cancelar" único (correção). Concluir/Falta não aparecem.
7. Tentar transição inválida via DevTools (modificar `to` no payload) → action retorna mensagem amigável de transição bloqueada.
