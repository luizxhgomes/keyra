# Story 3.1: Comanda Automática

## Status

Done

## Story

**Como** owner/profissional,
**Eu quero** ver e ajustar a comanda gerada automaticamente quando concluo um atendimento,
**Para que** o financeiro nasça da operação sem lançamento manual (CON-KE-02).

## Complexidade

**T-shirt:** S (~3 pontos · 0,5 dia)

## Contexto

O trigger `trg_appointments_done_to_command` já cria `commands` + `command_items` snapshotados quando o status do agendamento vira `done` (validado empiricamente em 2026-05-01). Esta story só **expõe** as comandas na UI e permite edição manual (FR-CO-02): adicionar item extra (produto/serviço fora do agendamento) e aplicar desconto.

## Acceptance Criteria

### AC1 — Listagem `/comandas`

1. Página `/comandas` protegida por `requireRole('professional')`.
2. Lista paginada (20/pág) ordenada por `opened_at DESC` com colunas: paciente, profissional, total (numeric, formatBRL), status (badge colorido), data abertura.
3. Filtro por status (toolbar): Todas · Abertas · Finalizadas · Pagas.
4. Empty state amigável: "Sem comandas ainda. Conclua um atendimento na agenda para gerar a primeira."

### AC2 — Detalhe `/comandas/[id]`

1. Mostra cabeçalho com paciente/profissional/agendamento de origem (link para `/agenda?date=...`).
2. Lista `command_items` (description, quantity, unit_price, line_total).
3. Card "Pagamentos" placeholder (Story 3.2 ativa).
4. Botão "Editar comanda" abre form de edição (apenas se `status='open'`).

### AC3 — Edição manual (FR-CO-02)

1. Adicionar item extra: select de produto/serviço ativo + quantity + unit_price (pre-preenchido com `services.price`, mas editável).
2. Remover item (ícone lixeira) — bloqueado para o item gerado pelo trigger (marcado por `service_id IS NOT NULL AND created_at = command.opened_at` ou flag).
3. Aplicar desconto: campo `discount_amount` (numeric ≥ 0). Total recalcula em tempo real.
4. Ao salvar, recalcula `commands.subtotal` (SUM line_total) e `commands.total` (subtotal - discount_amount); mantém `commands.status='open'`.

### AC4 — Status `finalized`

1. Botão "Finalizar comanda" muda `status` de `open` para `finalized` (FR-CO-03). Bloqueia novas edições.
2. Apenas comandas `finalized` aceitam pagamento (Story 3.2).

## Dev Notes

### Decisões de implementação

1. **Zero cálculo na aplicação.** `commands.subtotal` é mantido pelo trigger `trg_command_items_recompute` (AFTER INSERT/UPDATE/DELETE em `command_items`); `commands.total` é `GENERATED ALWAYS AS (subtotal - discount_amount) STORED`; `command_items.total` é `GENERATED ALWAYS AS ((unit_price * quantity) - discount_amount) STORED`. App só faz INSERT/DELETE/UPDATE em itens e SET `discount_amount` na comanda — banco cuida do resto. Isso elimina drift e simplifica o código drasticamente.
2. **`addCommandItem` snapshota unit_cost/commission do serviço atual** (não do agendamento), porque um item adicional não vem da agenda — vem da comanda em si. Se o esteticista vender um sérum extra na hora, o custo é o atual do produto. Item gerado pelo trigger continua snapshotado do momento do agendamento (preserva histórico de comissão pactuada).
3. **Defesa em camadas para `status='open'`.** Cada mutação (addItem, removeItem, applyDiscount, finalize) faz um SELECT prévio do status e rejeita com mensagem amigável se a comanda não está aberta. RLS é primeira linha; check de status é segunda linha; UI desabilita controles é terceira linha.
4. **Filtro `?status=` na lista** usa `searchParams` validado por type guard (`isCommandStatus`). Inválido → fallback para `all`. Mantém URLs portáveis e bookmarkáveis.
5. **`finalizeCommand` exige `total > 0`** (sem itens não dá pra finalizar). Mensagem amigável.
6. **Validação de `discountAmount ≤ subtotal`** evita total negativo (que o CHECK constraint do banco também bloquearia, mas a mensagem amigável é melhor antes).
7. **`/agenda?event=` link no detalhe** — permite voltar ao agendamento original. Implementação do query param `event` fica para uma sub-task quando precisarmos abrir o sheet automaticamente.
