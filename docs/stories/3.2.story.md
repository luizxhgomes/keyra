# Story 3.2: Registro de Pagamento

## Status

Ready

## Story

**Como** owner/profissional,
**Eu quero** registrar pagamento de uma comanda finalizada (Pix, cartão crédito/débito, dinheiro),
**Para que** a transação financeira seja gerada automaticamente e o lucro do atendimento entre no DRE (FR-FI-01).

## Complexidade

**T-shirt:** M (~5 pontos · 1 dia)

## Contexto

Schema já tem `payments`, `payment_methods` (com `fee_percent` e `default_settle_days`), `accounts` e `transactions`. O trigger `trg_payments_to_transaction` já cria a `transactions` (1:1 com payment), atualiza `commands.paid_amount`/`status`/`paid_at` e dispara o rateio de insumos via `_consume_command_inventory` (validado empiricamente). Esta story registra o pagamento na UI e expõe o cálculo de taxa.

## Acceptance Criteria

### AC1 — Form de pagamento em `/comandas/[id]`

1. Card "Pagamentos" no detalhe da comanda lista pagamentos existentes (forma, valor bruto, taxa, líquido, data).
2. Botão "Registrar pagamento" (visível apenas se `status='finalized'` e `paid_amount < total`).
3. Form: `payment_method_id` (Select de payment_methods ativos), `account_id` (Select de accounts), `gross_amount` (default = restante = `total - paid_amount`), `paid_at` (default = agora).

### AC2 — Cálculo automático de taxa (FR-CO-05)

1. Ao escolher payment_method, preview mostra: `fee_amount = gross_amount × fee_percent`, `net_amount = gross_amount - fee_amount`, `settle_at = paid_at + default_settle_days`.
2. Pix/dinheiro: fee=0, settle=imediato. Cartão crédito: fee=3.5%, settle=30d (defaults configuráveis).
3. Valores arredondados com `ROUND_HALF_EVEN` (NFR-FI-01) — Decimal.js no boundary.

### AC3 — Submissão dispara o trigger

1. Server Action `registerPayment` faz INSERT em `payments` com gross/fee/net e settle_at calculados; trigger `trg_payments_to_transaction` cria `transactions` (FR-FI-01) na mesma transação.
2. Toast: "Pagamento de R$ X registrado. Comanda atualizada."
3. Se `gross_amount + paid_amount >= total`, `status` da comanda vira `paid` automaticamente (trigger).
4. Refetch do detalhe: pagamentos aparecem; status da comanda atualiza.

### AC4 — Split de pagamento (parcial)

1. MVP suporta múltiplos pagamentos na mesma comanda (ex: R$ 100 Pix + R$ 200 cartão), cada um com seu fee. Cada pagamento dispara seu próprio trigger.
2. UI mostra `paid_amount` corrente e "restante" = `total - paid_amount`. Form pré-preenche com restante.

## Riscos

- **Taxa hardcoded vs configurável**: MVP usa `payment_methods.fee_percent` do banco (configurável). Defaults vêm do seed; se Luiz quiser mudar, edita em `/configuracoes/formas-pagamento` (sub-página opcional desta story).
- **Decimal.js obrigatório no cálculo**: `gross × fee_percent` não pode usar Number diretamente (ROUND_HALF_EVEN). Validar com test fixture: R$ 100,00 × 3,5% = R$ 3,50 (não 3.49999).
- **Idempotência**: dois clicks rápidos no botão "Registrar pagamento" não podem criar 2 payments. Usar `useTransition` + disabled durante pending.

## Definition of Done

1. Pagamento registrado dispara transação automática e atualiza comanda.
2. Cálculo de taxa correto para cada método (Pix=0, cartão=3.5%, etc).
3. Split de pagamento funciona (múltiplos payments na mesma comanda).
4. typecheck + lint + build verdes.
