# Story 4.9: Alertas Operacionais

## Status

Ready

## Story

**Como** owner,
**Eu quero** receber alertas visuais no dashboard quando algo importante muda (queda de lucro, baixa margem, alta taxa de faltas),
**Para que** eu reaja rápido sem precisar olhar relatórios todo dia (FR-IN-06).

## Complexidade

**T-shirt:** M (~5 pontos · 1 dia)

## Contexto

Não há motor de alertas separado no MVP — alertas são queries on-read no dashboard, sem persistência. Phase 5+ pode mover para tabela `alerts` com cron + dismissal. MVP entrega 4 regras codificadas.

## Acceptance Criteria

### AC1 — 4 regras de alerta

1. **Queda de lucro**: lucro do mês corrente < 80% do lucro do mês passado.
2. **Margem baixa**: margem do mês < 15% (lucro / receita).
3. **Alta taxa de faltas**: no_show + cancelled > 25% dos agendamentos do mês.
4. **Estoque baixo**: ≥ 3 insumos com `current_stock <= reorder_level`.

### AC2 — Card "Alertas" no dashboard

1. Visível apenas se houver ≥ 1 alerta ativo.
2. Cada alerta mostra: ícone (lucide-react `AlertTriangle`), texto descritivo, link para a página relevante.
3. Cores: âmbar (atenção) ou vermelho (crítico — apenas queda de lucro > 20%).

### AC3 — Sem persistência neste MVP

1. Alertas são calculados a cada render do dashboard.
2. Não há "dismiss" / "ler alerta" / histórico — apenas estado atual.

### AC4 — Performance

1. Cada regra é 1 query agregada simples; total ≤ 4 round-trips no Server Component.

## Riscos

- **Falsos positivos em mês novo**: regra "queda de lucro" precisa de pelo menos 1 mês de histórico; primeiro mês da clínica não dispara o alerta (guard `previousMonth_lucro > 0`).
- **Phase 5 vai querer notificações por email/WhatsApp**: schema atual é só on-read; expansão futura adiciona tabela `alerts` + Inngest.

## Definition of Done

1. 4 regras avaliadas a cada render.
2. Card aparece apenas com alertas ativos; some quando todos resolvidos.
3. Cores semânticas (âmbar/vermelho).
4. typecheck + lint + build verdes.
