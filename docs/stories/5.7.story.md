## Story

**Como** Camila (persona — dona de clínica de estética em uso intenso),
**Eu quero** que a linguagem visual nova (EmptyState, ErrorMessage, StatusBadge, FAB clicável) apareça em **todas** as telas onde antes havia `<p>` plano, `Erro: {error}` cru ou botão morto,

## Contexto

A Sprint 5 entregou a coluna vertebral (`EmptyState`, `ErrorMessage`, `Skeleton`, `loading.tsx`/`error.tsx`, tokens, StatusBadge universal, Button h-11, `--muted-foreground` AA) mas a **cobertura escapou** em ~14 lugares — auditoria estática do `baziotti` em `docs/ux/audit-2026-05-01-sprint5-review.md` (score 6.8/10) catalogou cada um.

## Dev Notes

### AC4.2 — pattern "adjust state during render"

Primeira tentativa usou `useEffect` para detectar mudança em `novoParam`, mas o React 19 + ESLint regra `react-hooks/set-state-in-effect` bloqueia setState síncrono em effect. Refatorado para o pattern recomendado pela [docs do React](https://react.dev/learn/you-might-not-need-an-effect#adjusting-state-when-a-prop-changes):

```tsx
const [prevNovoParam, setPrevNovoParam] = useState(false);
if (prevNovoParam !== novoParam) {
  setPrevNovoParam(novoParam);
  if (novoParam) {
    setFormInitialStartsAt(null);
    setFormOpen(true);
  }
}
```

Inicializa com `false` para que o primeiro render com `?novo=1` na URL dispare a abertura. `handleFormOpenChange` limpa o param via `router.replace(...)` ao fechar, permitindo que tocar `+` novamente regere a navegação e reabra o sheet.

### AC3 — decisão sobre `<AlertCard>`

Avaliado: o `<AlertCard>` existente em `components/keyra/` tem API distinta (single-alert, com `severity` enum) e o `AlertasCard` (lista de alertas) renderiza múltiplos itens dentro de `<ul>`. Adotar `<AlertCard>` exigiria refactor da API. Decisão: **opção (a) do AC3.2** — manter estrutura atual, trocar `border-blue-500` por `border-info` (token semântico KEYRA `#8A7A6B` já existente em `tailwind.config.ts:87`) e extrair `SEVERITY_ICON` map. Refactor profundo do `<AlertCard>` fica para Sprint 6.3 (alert hierarchy 3+5+5).
