# Story 6.1: Tipografia editorial — Inter Variable + escala expandida

## Status

Done

## Story

**Como** Camila (persona — dona de clínica de estética),
**Eu quero** que o KEYRA pareça uma ferramenta editorial sofisticada (não um app SaaS genérico) já no primeiro segundo de uso,
**Para que** eu sinta que estou usando uma plataforma à altura da minha clínica — uma marca confiável que respeita meu trabalho.

## Complexidade

**T-shirt:** M (~5 pontos · 1 dia)

## Contexto

A Sprint 5 fechou higienização funcional (StatusBadge, EmptyState, ErrorMessage, loading/error routes, tokens de spacing, AA). O que faltou para o produto **parecer profissional** é a **camada tipográfica** — atualmente 100% Inter regular nos pesos 400/500/600/700 sem hierarquia editorial.

Pesquisa do `@brand-chief` no Gestek (concorrente direto) revelou: **uma única família tipográfica com weights extremos** (200 para títulos display, 600/700 para botões e KPIs) cria voz editorial sem aumentar bundle. Princípio confirmado em [`docs/research/2026-05-01-gestek-design-system/`](../research/2026-05-01-gestek-design-system/).

Decisão (memory `feedback_keyra_visual_direction.md`): **Inter Variable**, não Fraunces serif (descartada após Gestek). Variable fonts entregam toda a faixa 100–900 num único arquivo (~30KB compactado vs ~80KB de 4 weights estáticos).

Esta story é a **base visual da Sprint 6**. Stories 6.2–6.5 (motion, alert hierarchy, DRE refactor, sidebar+FAB) consomem os tokens definidos aqui. Inverter a ordem força refactor.

## Acceptance Criteria

### AC1 — Inter Variable carregada

Substituir `Inter` (4 weights estáticos) por **Inter Variable**:

```ts
// apps/web/src/app/layout.tsx
import { Inter } from 'next/font/google';

const inter = Inter({
  subsets: ['latin'],
  variable: '--font-inter',
  display: 'swap',
  // Variable font — toda a faixa 100-900 num arquivo
});
```

`tailwind.config.ts` já referencia `--font-inter` (ou referenciar via `fontFamily.sans`). Confirmar.

### AC2 — Escala tipográfica de 10 tokens

Adicionar em `apps/web/tailwind.config.ts` extend.fontSize:

```ts
fontSize: {
  // Display (títulos editoriais — pesos finos com tracking negativo)
  'display-hero': ['56px', { lineHeight: '1.1',  letterSpacing: '-0.02em',  fontWeight: '200' }],
  'display':      ['40px', { lineHeight: '1.15', letterSpacing: '-0.015em', fontWeight: '200' }],
  // Headings (substantivos institucionais — peso semibold)
  'h1':           ['32px', { lineHeight: '1.2',  letterSpacing: '-0.01em',  fontWeight: '600' }],
  'h2':           ['24px', { lineHeight: '1.25', letterSpacing: '0',        fontWeight: '600' }],
  'h3':           ['20px', { lineHeight: '1.3',  letterSpacing: '0',        fontWeight: '600' }],
  // KPI (números do dashboard — pesos altos com tabular-nums)
  'kpi-hero':     ['56px', { lineHeight: '1',    letterSpacing: '-0.02em',  fontWeight: '600' }],
  'kpi':          ['40px', { lineHeight: '1',    letterSpacing: '-0.015em', fontWeight: '600' }],
  // Body / label (corpo legível)
  'body':         ['16px', { lineHeight: '1.5',  letterSpacing: '0',        fontWeight: '400' }],
  'body-sm':      ['14px', { lineHeight: '1.5',  letterSpacing: '0',        fontWeight: '400' }],
  'label':        ['12px', { lineHeight: '1.2',  letterSpacing: '0.05em',   fontWeight: '600' }],
}
```

> Nota 1: `text-label` deve ser combinado com `uppercase` no markup (não no token — Tailwind 3 não aceita `textTransform` no fontSize).
>
> Nota 2: a sintaxe `[size, { lineHeight, letterSpacing, fontWeight }]` é válida no Tailwind 3.4+ — define o `fontWeight` **default** quando `text-display-hero` é aplicado sozinho. Se outra classe (`font-bold`, `font-medium`) for aplicada junto, ela vence. Sem necessidade de `extend.fontWeight` separado.

### AC3 — Aplicação dirigida (não global)

Aplicar tokens cirurgicamente. **Não** trocar todos os `text-3xl`/`text-2xl` do codebase de uma vez — risco de regressão visual. Lugares obrigatórios:

| Arquivo | Antes | Depois |
|---------|-------|--------|
| `dashboard/page.tsx` (H1 "Dashboard") | `text-3xl font-bold tracking-tight` | `text-display-hero` |
| Outras páginas H1 (agenda, comandas, etc.) | `text-2xl font-bold tracking-tight` (média atual) | `text-display` |
| `KPICard.tsx` variant=`hero` (valor) | `text-4xl font-bold` (atual) | `text-kpi-hero` + `tabular-nums` mantido |
| `KPICard.tsx` variant=`secondary` (valor) | `text-3xl font-bold` (atual) | `text-kpi` + `tabular-nums` mantido |
| `Card.tsx` `CardTitle` shadcn | `text-lg font-semibold` (atual) | `text-h3` |
| Labels em uppercase tracking-wide (ex.: `SummaryCell`) | `text-xs uppercase tracking-wide` | `text-label uppercase` |

### AC4 — Buttons com weight semibold

`apps/web/src/components/ui/button.tsx` — variantes default e secondary sobem para `font-semibold` (600). Modelando Gestek (700) ajustado para sobriedade editorial. Tamanho permanece `h-11` (44px AA da Sprint 5).

### AC5 — Validação concreta de legibilidade

Antes de marcar Done:

1. **Display weight 200** — testar em mobile real (375px viewport) que o título "Dashboard" não fica fino demais. Critério: leitura confortável a 30cm de distância.
2. **Body 16px line-height 1.5** — passa AA contraste com `--foreground` sobre `--background` (já é 7+:1, herdado da Sprint 5.4). Confirmar.
3. **KPI value 56px weight 600 com `tabular-nums`** — todos os dígitos alinham coluna a coluna em todos os 4 KPIs do dashboard hero. Sem "pulo" entre 1, 2, 3.
4. **Bundle size** — comparar `Inter Variable` (1 arquivo) vs `Inter weights estáticos` (4 arquivos). Esperado: redução. Registrar números no QA Results.
5. **Greps de uso obrigatório** após implementação:
   - `text-display-hero` deve aparecer em `dashboard/page.tsx` (≥1 match).
   - `text-kpi-hero` deve aparecer em `KPICard.tsx` (≥1 match).
   - `text-h3` deve aparecer em `Card.tsx` (CardTitle, ≥1 match).

### AC6 — Validação de jornada de usuário (J1–J6)

Após implementação, validar mentalmente as 6 jornadas em `docs/ux/user-journeys.md`:

- **J1 (Onboarding)**: H1 "Dashboard" com `text-display-hero` é a primeira impressão. Camila percebe sofisticação editorial vs SaaS genérico.
- **J2 (Dia típico mobile)**: KPI hero `text-kpi-hero` legível em 375px sem zoom. Viewport mobile não quebra escala.
- **J3 (Fechamento de mês)**: tabelas DRE com `text-body-sm` para densidade; `text-h3` em CardTitle de cada DRE; números com `tabular-nums` alinhados.
- **J4 (Convidando profissional)**: forms `/team/convites` legíveis, labels em `text-label` uppercase, body em `text-body`.
- **J5 (Alerta de estoque)**: `<AlertasCard>` títulos com `text-h3`, descrições com `text-body-sm`. Mantém legibilidade quando borda lateral é sálvia/âmbar/destructive.
- **J6 (Cancelamento/falta)**: comanda detail com hierarquia visual clara (CardTitle → items → totals).

## Riscos

- **Display weight 200 fica fino demais em telas baixa-densidade** — mitigação: AC5.1 valida visualmente antes de Done. Se ficar fino, subir para 300 ou 400.
- **Inter Variable bundle pode não reduzir** se Next.js não otimizar tree-shaking de variable fonts — mitigação: AC5.4 mede e registra. Se aumentar significativamente, registrar como tech debt e considerar self-host com `font-display: swap`.
- **`text-h3` em CardTitle pode quebrar layouts existentes** que assumiam `text-lg` (18px) — agora vira 20px. Mitigação: testar com `pnpm build` + smoke das telas-chave (dashboard, comandas, financeiro/dre).
- **`textTransform: uppercase` não aceito em fontSize do Tailwind 3** — confirmado na nota do AC2. Aplicação via classe utility `uppercase` no markup. Se for esquecido, label fica em case original — não quebra build, só visual.

## Dev Notes

### AC1 — `axes: ['opsz']` testado e descartado

Primeira tentativa adicionou `axes: ['opsz']` para optical sizing automática. Build mediu:
- **Antes (sem opsz)**: 224KB total / 7 arquivos / maior subset 85KB
- **Com opsz**: 348KB total / 7 arquivos / maior subset 133KB
- **Diferença**: +124KB (~55% maior)

Tradeoff perceptual em escalas KEYRA (12px label → 56px display-hero) não compensa. Inter wght-only cobre toda a hierarquia editorial 200–600 sem custo extra. Bundle final voltou a 224KB (paridade exata com a Sprint 5).

### AC3 — sweep amplo dos labels

Plan inicial listava só 6 SummaryCells. O perl batch via regex `text-xs uppercase tracking-wide` pegou 14 ocorrências reais — incluindo `event-detail-sheet.tsx`, `agendamento-form.tsx` (3×), `dropdown-menu.tsx` (shadcn) e `error.tsx`. Decisão: manter o sweep amplo. Token `text-label` é definição de "rótulo uppercase" — qualquer lugar que era `text-xs uppercase tracking-wide` é candidato natural. Reduz inconsistência futura.
