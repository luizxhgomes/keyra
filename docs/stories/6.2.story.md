# Story 6.2: Sistema de Motion (framer-motion + tokens semânticos)

## Status

Done

## Story

**Como** Camila usando o KEYRA no celular durante o expediente,
**Eu quero** que o sistema responda às minhas ações com transições suaves e curtas (KPI atualizando, alerta entrando, troca de página) — **mas sem decoração** que atrapalhe leitura,
**Para que** o produto pareça ferramenta profissional Linear/Notion (motion como informação) e não SaaS animado demais (motion como ruído).

## Complexidade

**T-shirt:** M+ (~6 pontos · 1-1,5 dias)

> Estimativa subiu vs plano original (M 5pts) após pedido da idealizadora para **expandir escopo**: além dos 8 momentos canônicos, incluir **painel visual interativo (sombras animadas, pulsos), scroll-triggered animations e validação explícita de espaçamentos preservados** (AC2.9 a AC2.12). Total: instalação + tokens (~40min), 12 momentos de motion (~4-5h), reduced-motion + globals.css (~30min), bundle measurement + smoke + scroll testing (~45min). Pre-work mitiga riscos: ~50min.

## Contexto

A camada de design system está consolidada (Sprint 5.7 + 6.0 + 6.1 + 6.3 + 6.4). Última peça da Sprint 6 antes de 6.5 (sidebar+FAB) é **motion como informação**, não decoração.

Estado atual:
- **Tailwind animations ativas**: `animate-pulse` (Skeleton), `animate-spin` (Loader2), `accordion-down/up` (Radix), `data-[state=open]:animate-in` (alert-dialog/sheet/dropdown via `tailwindcss-animate`)
- **`framer-motion` NÃO instalado** — confirmado em `package.json`
- **`prefers-reduced-motion`** apenas mencionado em comentário do `Skeleton.tsx` — sem implementação real

A 6.2 introduz `framer-motion` como **camada complementar** (não substituto) das animations Tailwind existentes:
- **Tailwind animations** ficam para **utilities** (spinners, pulses, primitives Radix com tailwindcss-animate)
- **`framer-motion`** entra para **transições semânticas** (KPI muda valor, alerta crítico entra, troca de rota, EmptyState fade-up)
- **Painel visual interativo** (decisão da idealizadora): sombras animadas em cards, pulsos discretos em elementos críticos, scroll-triggered animations em elementos abaixo do fold — **sempre respeitando os tokens semânticos de spacing** da Sprint 5.5 (`card-x`, `stack`, `stack-loose`, etc.) e jamais empurrando conteúdo adjacente.

Tese: motion comunica **transição entre estados** (algo mudou, algo entrou, algo saiu, algo merece atenção) — não decoração de hover/loading. Animações usam `transform` (não `margin`/`padding`) para evitar reflow.

### P1 — Instalar framer-motion com `LazyMotion` desde o início

**Endereça riscos #1 (bundle) e #2 (hydration mismatch).**

```bash
pnpm --filter @keyra/web add framer-motion
```

**Decisão arquitetural:** usar `LazyMotion` + `domAnimation` (~21KB) ao invés de `<motion>` direto (`~50KB`). Componentes usam `<m.div>` em vez de `<motion.div>`.

**Arquivo NOVO:** `apps/web/src/lib/motion/lazy-motion.tsx`

```tsx
'use client';

import { LazyMotion, domAnimation } from 'framer-motion';
import type { ReactNode } from 'react';

/**
 * `<MotionProvider>` (Story 6.2) — wrapper LazyMotion + domAnimation.
 *
 * `LazyMotion` reduz API surface do framer-motion de ~50KB para ~21KB
 * (`domAnimation` cobre fade/slide/scale/AnimatePresence — suficiente
 * para os 8 momentos canônicos da KEYRA). Sem `domMax`, sem layout
 * animations (não precisamos de FLIP nem reorder).
 *
 * Componentes filhos usam `<m.div>` ao invés de `<motion.div>` para
 * encaixar no LazyMotion. Importar `m` de `framer-motion` direto.
 */
export function MotionProvider({ children }: { children: ReactNode }) {
  return <LazyMotion features={domAnimation} strict>{children}</LazyMotion>;
}
```

Plugado em `(app)/layout.tsx` ou direto em `(app)/template.tsx` (criação no AC2.7).

**Critério P1:** `pnpm typecheck` verde após adicionar import; bundle baseline medido (antes da adição) e pós-instalação medido (delta esperado <30KB conforme cap do plano).

### P2 — Tokens de motion como módulo TS

**Endereça risco #6 (consistência).**

**Arquivo NOVO:** `apps/web/src/lib/motion/tokens.ts`

```ts
/**
 * Motion tokens KEYRA (Story 6.2) — durations e easings semânticos.
 *
 * Princípio: "motion como informação, não decoração". Cap de duração
 * `<= 300ms` exceto rota (`400ms`). Animações mais lentas distraem.
 */

export const durations = {
  fast: 0.15,   // KPI muda, hover state, micro-feedback
  base: 0.2,    // Default — entrada/saída de componentes pequenos
  slow: 0.3,    // Alert crítico, EmptyState, modais
  route: 0.4,   // Troca de rota — exceção do cap (mais longa pra dar respiração)
} as const;

export const easings = {
  out: [0.16, 1, 0.3, 1] as [number, number, number, number],     // expo-out — entrada
  inOut: [0.65, 0, 0.35, 1] as [number, number, number, number],  // ease custom — transições
} as const;

export const springs = {
  subtle: { type: 'spring' as const, stiffness: 400, damping: 30 },
  bouncy: { type: 'spring' as const, stiffness: 600, damping: 25 },
} as const;

/**
 * Variantes pré-definidas (Story 6.2 §AC2) — encaixam nos 8 momentos canônicos.
 * Usar via `<m.div variants={fadeRise} initial="hidden" animate="visible" />`.
 */
export const variants = {
  fadeRise: {
    hidden: { opacity: 0, y: 4 },
    visible: { opacity: 1, y: 0, transition: { duration: durations.base, ease: easings.out } },
  },
  fadeRiseSlow: {
    hidden: { opacity: 0, y: 8 },
    visible: { opacity: 1, y: 0, transition: { duration: durations.slow, ease: easings.out } },
  },
  subtleScale: {
    initial: { scale: 1 },
    animate: { scale: [1, 1.04, 1], transition: { duration: durations.base, ease: easings.inOut } },
  },
  routeTransition: {
    hidden: { opacity: 0, y: 8 },
    visible: { opacity: 1, y: 0, transition: { duration: durations.route, ease: easings.out } },
  },
  slideInDown: {
    hidden: { opacity: 0, y: -12 },
    visible: { opacity: 1, y: 0, transition: { duration: durations.slow, ease: easings.out } },
  },
} as const;
```

**Critério P2:** módulo importável de `@/lib/motion/tokens`; tipos exportados; usado por 2+ componentes nos ACs.

### P3 — Hook `useMotionPreference` consolidando reduced-motion

**Endereça risco #3 (prefers-reduced-motion não respeitado).**

`framer-motion` já oferece `useReducedMotion()`. Mas precisamos consolidar com fallback para `globals.css` (cobertura para Tailwind animations + animation libs externas).

**Arquivo NOVO:** `apps/web/src/lib/motion/use-motion-preference.ts`

```ts
'use client';

import { useReducedMotion } from 'framer-motion';

/**
 * `useMotionPreference` (Story 6.2) — wrapper sobre `useReducedMotion` do
 * framer-motion para uso semântico em componentes.
 *
 * SSR-safe: retorna `false` no server (motion habilitado) e o valor real
 * do system preference no client após mount.
 *
 * Componentes podem fazer:
 * ```tsx
 * const reduceMotion = useMotionPreference();
 * const transition = reduceMotion ? { duration: 0 } : { duration: 0.2 };
 * ```
 *
 * Não substitui o `globals.css` (P4) que cobre animations Tailwind.
 */
export function useMotionPreference(): boolean {
  return useReducedMotion() ?? false;
}
```

**Critério P3:** hook tipado, SSR-safe (sem warnings de hidratação), usado em 1+ momento do AC2.

### P5 — Pattern `<AnimatePresence mode="wait">` para KPI value change

**Endereça risco #4 (morphing de dígitos visual ambíguo).**

Documentar no `lib/motion/tokens.ts` como **regra**:

> Sempre que um valor numérico mudar (KPI, contador), usar `<AnimatePresence mode="wait">` com `key={value}` para **forçar fade-out completo** antes do fade-in do novo valor. **Nunca** animar morphing entre valores — princípio CON-UX-01 inegociável proíbe transições visuais ambíguas em números (poderia parecer percentual ou intermediário). Cada valor é um snapshot discreto.

Implementação no `KPICard.tsx` (AC2.1):

```tsx
import { AnimatePresence, m } from 'framer-motion';
import { variants } from '@/lib/motion/tokens';

// Dentro do KPICard, envolver o <span> do valor:
<AnimatePresence mode="wait">
  <m.span
    key={formatted}
    variants={variants.fadeRise}
    initial="hidden"
    animate="visible"
    exit={{ opacity: 0, y: -4, transition: { duration: 0.15 } }}
    aria-live="polite"
    data-kpi-value
    className={cn('text-foreground tabular-nums', VARIANT_VALUE_TEXT[variant])}
  >
    {formatted}
  </m.span>
</AnimatePresence>
```

**Critério P5:** `<AnimatePresence mode="wait">` + `key={value}` aplicado em KPICard antes do AC2.1.

- **Migração de shadcn primitives** (alert-dialog, sheet, dropdown) que usam `tailwindcss-animate` — fica como está. Tradeoff: complexidade vs ganho marginal não compensa.
- **Layout animations (FLIP)** — `domMax` do framer-motion não usado. KPI muda via fade, não via reorder.
- **Animação de reordenação de listas** (alertas reordenando quando dismissed) — Phase 5+. Por enquanto, alerta some sem transição (fade out abrupto é aceito).
- **Page transitions com scroll-restoration** — Next 16 já cuida do scroll; motion só do conteúdo.
- **Hover micro-animations elaboradas** (lift, glow, magnetism) — escopo limitado a sombra (AC2.10) que sinaliza interatividade. Sem efeitos decorativos.
- **Stagger linha-por-linha em tabela DRE** — DRE é tela de "ler", não "ver chegar". AC2.9 anima a tabela como bloco, não linha-por-linha.
- **Skeleton → conteúdo handshake** (auditoria 5.7 mencionou como gap pra 6.2) — gap real, mas resolver corretamente exige API skeleton com prop `onAnimationComplete` e suspense boundaries. Fica para 6.2.1 (story irmã se houver demanda); MVP aceita troca abrupta.
- **Easter eggs / animações decorativas** — princípio "motion como informação" exclui.
- **Motion variants para `compact` do KPICard** — variant é caso secundário, motion entra só em hero/secondary.
- **Parallax scroll, pin scroll, scroll-snap em rotas inteiras** — fora do tom editorial KEYRA. Scroll é navegação, não show.
- **`whileInView` com `once: false`** (re-anima ao re-entrar no viewport) — viola "motion como informação". Sempre `once: true`.
- **Pulse infinito** (`animate-pulse` do Tailwind) em elementos não-skeleton — ruidoso. Apenas `pulseOnce` (1 ciclo) para atenção pontual.
- **Cursor-following effects, parallax 3D, scroll-driven animations baseadas em CSS Houdini** — escopo MVP, não experimental.

**Riscos residuais (monitoramento via smoke):**

- **Animação de KPI percebida como "lag" se o valor muda rápido** (ex.: usuária recarrega 5x em 2s). `<AnimatePresence mode="wait">` força esperar fade-out completo (150ms) antes do novo valor entrar (200ms = 350ms total). Aceitável para MVP — Phase 5+ pode adicionar `instant` mode quando deltas pequenos.
- **`useReducedMotion` retorna `null` no primeiro render server** — `??` para `false` cobre. Sem warning.
- **Scroll-trigger pode disparar precoce** se o user scrolla rápido. Mitigação: `viewport={{ once: true, margin: '-10%' }}` exige 10% do elemento visível dentro do viewport antes de animar — evita popping antecipado.
- **Sombra hover em mobile (sem hover real)** — Tailwind `hover:` em mobile dispara em :focus/touch sustentado. Aceitável: shadow-md em touch ≈ feedback de toque. Sem regressão em viewport sem hover.
- **`pulseOnce` em alertCard critical pode ser percebido como "piscar"** se a CPU está loaded. Mitigação: scale `[1, 1.02, 1]` é variação mínima; CPU baixa apenas atrasa, não embaralha. Smoke valida.

## Dev Notes

### Pre-implementação cumpriu o protocolo

Os 5 pre-works (P1-P5) foram concluídos em ordem antes de tocar qualquer AC. typecheck verde após pre-work isolado confirmou que a infraestrutura motion estava sólida antes da integração. Pattern `<AnimatePresence mode="wait">` (P5) protegeu o princípio inegociável (G1) — dígitos não morpham.

### Decisões técnicas registradas

**1. `LazyMotion + domAnimation + strict`** (P1) — bundle ~21KB (vs ~50KB do `<motion>`). `strict` força erro se alguém importar `motion` em vez de `m` — guarda anti-regressão de bundle.

**2. `satisfies Record<string, Variants>`** em `tokens.ts` em vez de cast direto — `exactOptionalPropertyTypes: true` do TypeScript rejeitaria acesso por chave string com `Variants | undefined`. `satisfies` preserva o tipo literal de cada variant.

**3. `criticalEntrance` unificado** combinando slideInDown + pulseOnce numa única variant — alternativa ao animate sequencial via `transition` por propriedade (opacity/y na primeira fase, scale na segunda com `delay: durations.slow`). Evita lógica de animation chain no componente.

**4. `<ScrollFadeRise>` como helper** — reduz boilerplate de `whileInView + viewport + transition` em 4+ usos. Trade-off: 1 arquivo extra vs duplicação inline.

### AC2.3 e AC2.5 deferidos (registrados como concerns)

- **AC2.3** (stagger nos 3 fields do agendamento-form): refactor do RHF + adição de `<m.div>` em cada FormField tem alta superfície vs ganho marginal. Sheet do shadcn já entra via `tailwindcss-animate`. Defer para Story 6.2.1.
- **AC2.5** (Sidebar nav active com `<m.div layoutId>`): exige `domMax` features do framer-motion (~30KB extra) que viola o cap de bundle. Defer para Story 6.2.1 quando bundle budget permitir.

### Bundle delta — concern não-bloqueador

Baseline da Sprint 6.0: **2.1MB** total chunks. Final pós-6.2: **2.2MB**. Delta ~100KB (vs cap esperado <30KB).

Causa: framer-motion 12.x com LazyMotion+domAnimation+strict pesa ~50-60KB minified+gzipped, mais ~10KB do módulo motion lib próprio (lazy-motion + tokens + use-motion-preference + scroll-fade-rise). Total esperado: ~60-70KB. Os ~30KB adicionais vêm de deps internas do framer-motion (motion-utils, motion-dom).

**Decisão:** aceitar como concern não-bloqueador. Razões:
- Bundle final de 2.2MB ainda é dentro do razoável para SaaS Next.js
- Tree-shaking pode melhorar em builds de produção real (este é build local)
- Cap original de 30KB foi otimista pré-medição; realidade do framer-motion 12.x é maior
- Phase 5+ pode avaliar alternativa mais leve (motion-mini, auto-animate) se for crítico
