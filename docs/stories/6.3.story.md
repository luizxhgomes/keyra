# Story 6.3: Alert hierarchy 3+5+5 + dismiss + fonte única

## Status

Done

## Story

**Como** Camila operando a clínica de manhã (J5 — alerta de estoque baixo),
**Eu quero** que os alertas no dashboard tenham uma hierarquia visual clara (crítico salta, aviso pondera, informativo respira) e que eu possa **silenciar** alertas que já estou tratando — sem que sumam para sempre,
**Para que** o dashboard não vire um pântano de avisos amarelos depois de 30 dias de uso e eu confie no sinal de "isto requer ação agora".

## Complexidade

**T-shirt:** M (~5 pontos · 1 dia)

## Contexto

A Story 5.7 (higienização de cobertura) deixou o `AlertasCard` com **duas fontes de verdade** convivendo no mesmo componente:

- `border-l-4` lateral com tokens semânticos (`border-info`, `border-amber-500`, `border-destructive`)
- `<StatusBadge>` (pill) com label textual ("Crítico", "Aviso", "Info")
- Ícone `AlertTriangle` único para todas as severidades

E o `<AlertCard>` (em `components/keyra/AlertCard.tsx`) **existe há tempos sem uso em produção** — design completo com `bg-amber-50`/`bg-red-50`/`bg-muted` cheio + ícones canônicos por severity (`AlertTriangle`/`AlertOctagon`/`Info`) + role ARIA correto. Foi planejado mas nunca adotado. A 5.7 manteve a estrutura antiga porque refactor profundo era out of scope.

Esta story **adota o `<AlertCard>` como fonte única**, implementa a regra **3+5+5** (cap por severidade — base para Phase 5 quando o sistema gerar mais que os 4 alertas atuais) e adiciona **dismiss via localStorage** ("Silenciar 7 dias") — a única feature realmente nova além do refactor.

> **Sobre o `<StatusBadge>`**: hoje o pill mostra "Crítico"/"Aviso"/"Info" — texto redundante com o `bg` + ícone do `<AlertCard>`. Vamos remover o pill nesta refatoração — o bg colorido + ícone canônico do `<AlertCard>` já comunicam severity sem ambiguidade. Acessibilidade preservada via `role="alert"` (critical) e `role="status"` (warning/info), já no `<AlertCard>`.

### AC2 — Regra 3+5+5 (cap por severity)

No componente que renderiza a lista de alertas, ordenar por severity (`critical` > `warning` > `info`) e aplicar caps:

```ts

// Para cada severity: alertas.filter(...).slice(0, CAPS[severity])
```

### AC3 — Dismiss inline ("Silenciar 7 dias")

Cada `<AlertCard>` ganha botão secundário "Silenciar 7 dias" ao lado da action principal "Investigar →".

**Persistência (MVP — localStorage):**
- Key: `keyra:dismissed-alerts:{orgId}`
- Valor: array de `{ alertId: string, dismissedUntil: string /* ISO 8601 */ }`
- Cleanup: na hidratação do componente client, remover entries onde `dismissedUntil < now` antes de filtrar
- TTL de 7 dias = `now + 7 * 24 * 60 * 60 * 1000` no momento do clique

**Filtro:**
- Após o cleanup, alertas com `id` em `dismissed[].alertId` (não-expirados) **não renderizam**
- Se TODOS forem dismissed: empty state diz "**N alertas silenciados** nesta semana" (variação calma do empty state da 5.7 — informativa, não positiva)

**Restrição de severity:**
- `critical` **não** pode ser silenciado. Botão "Silenciar 7 dias" oculto na severidade crítica. Justificativa: alerta crítico (queda de lucro > 20%) é o tipo que Camila não pode "esquecer por uma semana" — é exatamente nesses dias que ela mais precisa olhar.

**Server vs Client:**
- `AlertasCard` continua Server Component (busca dados via `getActiveAlerts`)
- Novo componente `<AlertasList>` (Client) recebe `alerts: Alert[]` + `orgId: string` e gerencia a lista filtrada
- Pattern "adjust state during render" (React 19 — Sprint 5.7 e 6.1 já usaram) para hidratação inicial sem `useEffect` que dispare setState

### AC4 — Empty state positivo (preservar + 1 variação)

A Story 5.7 já entregou empty state positivo quando 0 alertas. **Preservar exatamente** + adicionar uma variação:

| Cenário | Texto | Ícone |
|---------|-------|-------|

> Nuance importante: segundo cenário **não** é "tudo bem" — é "você decidiu adiar". Cor neutra (`text-muted-foreground` no ícone, não sálvia).

### P2 — Utility `safeLocalStorage`

**Endereça risco #5.**

**Arquivo NOVO:** `apps/web/src/lib/storage.ts`

```ts
/**
 * `safeLocalStorage` (Story 6.3) — wrapper SSR-safe e fail-safe para localStorage.
 *
 * Trata cenários onde localStorage falha:
 * - SSR (window não definido)
 * - Modo privado/incognito (Safari iOS lança QuotaExceededError em setItem)
 * - Storage cheio
 * - JSON parse de valor corrompido
 *
 * Retorna fallback ao invés de throw — UX permanece estável.
 */
export const safeLocalStorage = {
  getItem(key: string): string | null {
    if (typeof window === 'undefined') return null;
    try {
      return window.localStorage.getItem(key);
    } catch {
      return null;
    }
  },

  setItem(key: string, value: string): boolean {
    if (typeof window === 'undefined') return false;
    try {
      window.localStorage.setItem(key, value);
      return true;
    } catch {
      return false;
    }
  },

  removeItem(key: string): void {
    if (typeof window === 'undefined') return;
    try {
      window.localStorage.removeItem(key);
    } catch {
      // Silencioso — falha aqui não afeta UX.
    }
  },

  getJSON<T>(key: string, fallback: T): T {
    const raw = this.getItem(key);
    if (raw === null) return fallback;
    try {
      return JSON.parse(raw) as T;
    } catch {
      return fallback;
    }
  },

  setJSON<T>(key: string, value: T): boolean {
    try {
      return this.setItem(key, JSON.stringify(value));
    } catch {
      return false;
    }
  },
};
```

**Critério de aceite P2:** módulo importável de `@/lib/storage`; chamadas em SSR retornam `null`/`false`/fallback sem throw.

### P3 — Hook `useDismissedAlerts` com `useSyncExternalStore`

**Endereça riscos #1 e #2 simultaneamente.**

**Arquivo NOVO:** `apps/web/src/lib/hooks/use-dismissed-alerts.ts`

Pattern canônico React 18+ para sincronizar state cliente com fonte externa (localStorage). Sem `useEffect` mutativo (compliance React 19 `react-hooks/set-state-in-effect`). SSR retorna snapshot vazio, client retorna real após mount — sem hydration mismatch warning.

```ts
'use client';

import { useCallback, useSyncExternalStore } from 'react';

import { safeLocalStorage } from '@/lib/storage';

type DismissedEntry = { alertId: string; dismissedUntil: string };

const STORAGE_PREFIX = 'keyra:dismissed-alerts:';
const DISMISS_DAYS = 7;
const DISMISS_TTL_MS = DISMISS_DAYS * 24 * 60 * 60 * 1000;

function getStorageKey(orgId: string): string {
  return `${STORAGE_PREFIX}${orgId}`;
}

function readEntries(orgId: string): DismissedEntry[] {
  const raw = safeLocalStorage.getJSON<DismissedEntry[]>(getStorageKey(orgId), []);
  if (!Array.isArray(raw)) return [];
  // Cleanup de entries expiradas — feito on-read.
  const now = Date.now();
  return raw.filter((e) => {
    if (!e || typeof e.alertId !== 'string' || typeof e.dismissedUntil !== 'string') {
      return false;
    }
    const until = new Date(e.dismissedUntil).getTime();
    return Number.isFinite(until) && until > now;
  });
}

// Subscribe global a `storage` event — sincroniza entre tabs do mesmo browser.
function subscribe(callback: () => void): () => void {
  if (typeof window === 'undefined') return () => {};
  window.addEventListener('storage', callback);
  return () => window.removeEventListener('storage', callback);
}

export interface UseDismissedAlertsResult {
  dismissedIds: Set<string>;
  dismiss: (alertId: string) => void;
  isDismissed: (alertId: string) => boolean;
}

export function useDismissedAlerts(orgId: string): UseDismissedAlertsResult {
  // Server snapshot: array vazio (zero alertas dismissed em SSR).
  // Client snapshot: lê do localStorage com cleanup automático.
  const entries = useSyncExternalStore<DismissedEntry[]>(
    subscribe,
    () => readEntries(orgId),
    () => [],
  );

  const dismissedIds = new Set(entries.map((e) => e.alertId));

  const dismiss = useCallback(
    (alertId: string) => {
      const current = readEntries(orgId);
      const dismissedUntil = new Date(Date.now() + DISMISS_TTL_MS).toISOString();
      const next = [
        ...current.filter((e) => e.alertId !== alertId),
        { alertId, dismissedUntil },
      ];
      safeLocalStorage.setJSON(getStorageKey(orgId), next);
      // Dispara `storage` event para outros tabs + força re-render do hook.
      // useSyncExternalStore reescuta automaticamente.
      window.dispatchEvent(new StorageEvent('storage', { key: getStorageKey(orgId) }));
    },
    [orgId],
  );

  const isDismissed = useCallback(
    (alertId: string) => dismissedIds.has(alertId),
    [dismissedIds],
  );

  return { dismissedIds, dismiss, isDismissed };
}
```

**Critério de aceite P3:**
- typecheck verde
- SSR não dispara erro (subscribe/getServerSnapshot retornam vazio)
- Hidratação no client lê localStorage e popula `dismissedIds` no primeiro render — sem warning de hydration mismatch
- Click em "Silenciar" atualiza UI imediatamente (state via `useSyncExternalStore`)
- Storage event entre abas: silenciar em uma aba reflete em outra

## Dev Notes

### Pre-implementação reduziu o trabalho dos ACs

O pre-work (~50min: P1+P2+P3) eliminou os 5 riscos antes que se manifestassem. AC1-AC4 viraram **integração** dos artefatos prontos:

- AC1 não teve "duvida sobre como fazer" — `<AlertCard>` já tinha `secondaryAction` (P1), bastou usar.
- AC3 não teve "como persiste no localStorage" — `useDismissedAlerts` já gerenciava (P3).
- AC5 não teve "como SSR não quebra" — `useSyncExternalStore` resolveu na P3.

Custo do pre-work pago no AC: **AlertasCard** caiu de 60+ linhas para 26 (substituiu lógica complexa por delegação).

### Correção de lint pós-implementação

Lint pegou warning `react-hooks/exhaustive-deps`: `dismissedIds` como `new Set()` direto fazia a dependência do `useCallback isDismissed` mudar a cada render. Fix: envolver em `useMemo([entries])`. 2 linhas adicionadas, 1 modificada.

### Decisão sobre acessibilidade do botão Silenciar

`<button type="button">` com classe `text-sm text-muted-foreground hover:text-foreground` — sem `<Button>` shadcn por escolha visual deliberada: secondaryAction é **menos proeminente** que a action principal (Investigar). Touch target preservado por altura natural do texto + `gap-3` no flex parent (4px × 2 + 14px texto = ~22px clicável; é menor que AA 44px, mas é botão secundário em alerta — usuária toca conscientemente, não acidentalmente). Se smoke real revelar problema, escalar para `Button size="sm"` ou aumentar padding.

### `space-y-stack-loose` no `<ul>`

Token semântico (24px da Sprint 5.5) substitui `space-y-2` (8px) original. Decisão antecipada do P4. Cards `bg-amber-50`/`bg-red-50`/`bg-muted` empilhados em mobile 375px ganham respiração visual sem voltar ao `border-l-4`.
