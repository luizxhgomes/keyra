# Story 6.5: Sidebar 7 itens + FAB contextual por rota

## Status

Done

## Story

**Como** Camila usando o KEYRA,
**Eu quero** que a Sidebar (desktop) priorize as 7 áreas operacionais (Dashboard/Agenda/Comandas/Pacientes/Serviços/Financeiro/Estoque) **e** o FAB (mobile) leve para a ação mais natural daquela tela,
**Para que** eu não precise navegar dois cliques para criar um paciente quando estou em `/pacientes`, ou abrir agendamento manualmente sempre que clicar `+`.

## Complexidade

**T-shirt:** S (~3 pontos · 0,5 dia)

## Contexto

A Sidebar atual mistura **9 itens** num único bloco — 7 operacionais + Time + Configurações. Camila olha 9 ícones e perde a hierarquia "trabalho diário" vs "configuração". O plano da Sprint 6 (`docs/ux/PLANO-SPRINT-5-6.md` §6.5) reorganiza para 7 primários + 2 secundários separados por `border-t`.

O FAB do BottomNav (mobile) já navega via Story 5.7 para `/agenda?novo=1` — mas é estático: em `/pacientes`, tocar `+` força a usuária a sair, criar agendamento, voltar. Esta story torna o FAB **contextual por `pathname`** — em `/pacientes` o `+` cria paciente; em `/financeiro/despesas` cria despesa; etc.

Estado atual mapeado pelos gates Phase 2.5:

- **Sidebar.tsx**: 9 itens em `NAV_ITEMS`, layout único com `border-t` no rodapé contendo apenas texto auxiliar — refactor estrutural mínimo.
- **BottomNav.tsx**: FAB com `<Link href="/agenda?novo=1">` (Story 5.7). aria-label = "Criar novo agendamento" — fixo.
- **Tokens**: todos os tokens necessários existem (`border-l-4`, `border-primary`, `bg-primary-50`, `border-border`, `bg-muted`).

### AC2 — FAB do BottomNav contextual por `pathname`

`apps/web/src/components/layout/BottomNav.tsx` — substituir `<Link href="/agenda?novo=1">` fixo por uma função `getContextualAction(pathname)` que retorna `{ href, label }`:

| `pathname` começa com | `href` (destino) | `label` (aria-label) |
|----------------------|------------------|----------------------|
| `/pacientes` | `/pacientes/novo` | `Cadastrar novo paciente` |
| `/servicos` | `/servicos/novo` | `Cadastrar novo serviço` |
| `/financeiro/despesas` | `/financeiro/despesas/nova` | `Cadastrar nova despesa` |
| `/financeiro/metas` | `/financeiro/metas` | `Definir nova meta` *(form na própria página, scroll)* |
| `/comandas` | `/agenda?novo=1` | `Criar agendamento (gera comanda automática)` |
| `/estoque/insumos` | `/estoque/insumos/novo` | `Cadastrar novo insumo` |
| **default** (incluindo `/dashboard`, `/agenda`, `/financeiro`, `/estoque`, `/team`) | `/agenda?novo=1` | `Criar novo agendamento` |

Implementação como função pura para testabilidade futura:

```tsx
function getFabAction(pathname: string | null): { href: string; label: string } {
  if (!pathname) return { href: '/agenda?novo=1', label: 'Criar novo agendamento' };
  if (pathname.startsWith('/pacientes')) return { href: '/pacientes/novo', label: 'Cadastrar novo paciente' };
  if (pathname.startsWith('/servicos')) return { href: '/servicos/novo', label: 'Cadastrar novo serviço' };
  if (pathname.startsWith('/financeiro/despesas')) return { href: '/financeiro/despesas/nova', label: 'Cadastrar nova despesa' };
  if (pathname.startsWith('/estoque/insumos')) return { href: '/estoque/insumos/novo', label: 'Cadastrar novo insumo' };
  if (pathname.startsWith('/comandas')) return { href: '/agenda?novo=1', label: 'Criar agendamento (gera comanda automática)' };
  return { href: '/agenda?novo=1', label: 'Criar novo agendamento' };
}
```

> **Decisão sobre tooltip**: AC3 do plano original mencionava "tooltip no hover do FAB explicando ação contextual". **Reformulado como `aria-label` dinâmico** — BottomNav é mobile-only (`lg:hidden`), e mobile não tem hover. Tooltip Radix em viewport sem hover seria over-engineering. `aria-label` cobre acessibilidade (screen readers leem ao focar) e copy clara. **Tooltip explícito fica out of scope.**

### AC3 — Validação de jornada de usuário (J1–J6)

- **J1 (onboarding)**: Camila vai para `/pacientes` → cadastra primeiro paciente → toca FAB → `aria-label="Cadastrar novo paciente"` → vai direto para `/pacientes/novo`. Sem desvio pelo `/agenda`.
- **J2 (mobile dia típico)**: Camila em `/agenda` → toca FAB → abre sheet de novo agendamento (Story 5.7 + 6.2 com motion). Sem mudança de comportamento — preservado.
- **J3 (fechamento de mês)**: Camila em `/financeiro/despesas` → toca FAB → `/financeiro/despesas/nova`. Em `/financeiro/dre` (sem rota de criação) → fallback `/agenda?novo=1` (default).
- **J4 (convidando profissional)**: `/team` → fallback default `/agenda?novo=1`. Convites são feitos via página `/team/convites`, não via FAB. Aceitável — não é rota frequente.
- **J5 (alerta de estoque)**: `/estoque/insumos` → FAB cadastra insumo. `/estoque/movimentacoes` → fallback default (movimentações automáticas via comandas, sem CRUD manual nesta fase).
- **J6 (cancelamento/falta)**: `/agenda` → FAB cria agendamento (default).

**Sidebar AC1**: nas 6 jornadas, a usuária vê 7 itens primários no topo + separador + 2 secundários (Time, Configurações). Hierarquia visual clara — operação acima, configuração abaixo.

## Out of Scope

- **Tooltip Radix no FAB** — viewport mobile não tem hover; `aria-label` dinâmico cobre. Reformulação do AC3 do plano original.
- **Rota `/configuracoes`** — placeholder visual no NAV_SECONDARY; rota implementada em Phase 5+.
- **FAB que abre Sheet inline** (em vez de navegar para `/agenda?novo=1`) — exigiria portal global de Sheet + estado compartilhado. Story 6.2 já traz a entrada via query string da rota `/agenda` que abre o sheet automaticamente. Mantém-se.
- **Submenus / accordion na Sidebar** — `/financeiro` e `/estoque` têm sub-navegação dentro de cada layout próprio. Não centralizar.
- **Indicador animado de active na Sidebar (`<m.div layoutId>`)** — deferido na Story 6.2 (AC2.5) por exigir `domMax` features (+30KB) que viola cap de bundle. Permanece deferido para Story 6.2.1.
- **Reordenação personalizada da Sidebar** (drag&drop) — Phase 5+.
- **Pinning / favoritos** — Phase 5+.
- **Badges com contador** (ex.: "3 alertas") na Sidebar — fora do tema motion atual; estatística fica no dashboard.
- **Mais entradas no FAB contextual** (`/comandas/[id]` → registrar pagamento; `/agenda/[id]` → editar) — escopo MVP cobre rotas-índice; rotas detalhe ficam para Phase 5+.

## Riscos

- **Rota `/configuracoes` clicável mas inexistente** → 404 ao clicar. Comportamento idêntico ao da Sidebar atual (já era 9 itens com `/configuracoes`). Sem regressão. Documentado em Out of Scope.
- **Mapeamento contextual incompleto** — pathnames como `/team/convites` ou `/comandas/[id]` caem no fallback default (`/agenda?novo=1`). Aceitável; FAB é atalho, não cobertura total. Phase 5+ pode expandir.
- **`getFabAction` chamado em SSR** com `pathname=null` — guard `if (!pathname) return default` cobre. Sem warning de hidratação esperado.
- **Sidebar `border-t` empurra rodapé para cima quando viewport é curto** — `flex-1` no `<nav>` primário absorve o crescimento; secundário fica colado no fundo. Smoke valida em viewport 768×600 (laptop pequeno).
