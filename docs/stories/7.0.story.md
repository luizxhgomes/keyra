# Story 7.0: Habilitar Sentry capture no `(app)/error.tsx`

## Status

Done

## Story

**Como** orquestrador investigando o digest opaco `3213099672` reportado pela idealizadora em validação manual,
**Eu quero** que o error boundary global do `(app)/` envie todos os erros não-tratados para o Sentry com stacktrace completo,
**Para que** o root cause de qualquer erro futuro seja diagnosticável em produção em vez de ficar como digest opaco — e a Story 7.5 possa definir escopo baseado em evidência real.

## Complexidade

**T-shirt:** XS (~1 ponto · 30min)

## Contexto

A Story 5.2 (loading + error routes) entregou `(app)/error.tsx` com tom mentora confiável e captura de `error.digest` na UI, mas deixou comentário pendente:

```tsx
// apps/web/src/app/(app)/error.tsx:24-27
useEffect(() => {
  // TODO: Sentry capture quando configurado em produção  ⚠️ TODO ABERTO
  console.error('[KEYRA] Unhandled error:', error);
}, [error]);
```

**Sentry está instrumentado** desde Story 1.5 (ADR-021):
- `apps/web/src/instrumentation.ts` (Server)
- `apps/web/src/instrumentation-client.ts` (Client)

Mas o boundary global **não chama `captureException`** — então qualquer erro não-tratado dentro do route group autenticado fica opaco em produção.

**Validação manual em 2026-05-02** descobriu o impacto real: idealizadora caiu em "ALGO DEU ERRADO" com `digest 3213099672` após login válido. Stacktrace inacessível, debug impossível.

Esta story corrige o TODO em **1 linha de import + 1 linha de chamada** e desbloqueia investigação do root cause na Story 7.5.

## Acceptance Criteria

### AC1 — `Sentry.captureException(error)` no `useEffect`

`apps/web/src/app/(app)/error.tsx` — adicionar import e chamar capture:

```tsx
'use client';

import * as Sentry from '@sentry/nextjs';  // ← novo import
// ... outros imports

useEffect(() => {
  Sentry.captureException(error);  // ← nova linha
  console.error('[KEYRA] Unhandled error:', error);
}, [error]);
```

Manter `console.error` para debug local (dev). Sentry vai dedupe baseado em fingerprint.

### AC2 — Remover comentário TODO obsoleto

Apagar `// TODO: Sentry capture quando configurado em produção` — substituído por código real.

### AC3 — Verificações finais

- `pnpm typecheck` verde
- `pnpm lint --max-warnings 0` verde
- `pnpm build` verde (33 routes mantidas)
- `grep -n "captureException" apps/web/src/app/\(app\)/error.tsx` → ≥ 1 match
- `grep -rn "TODO.*Sentry" apps/web/src/app` → 0 matches
- **Smoke após deploy:** forçar erro em produção (visitar URL inválida ou injetar throw temporário) → verificar erro aparece no Sentry dashboard com stacktrace completo

## Out of Scope

- **Investigar root cause do digest 3213099672** — fica para Story 7.5 (após Sentry capturar evidência real)
- **Adicionar boundaries granulares** nos cards do dashboard — Story 7.5 AC2
- **Sentry para outros error boundaries** (ex.: `(auth)/`, `/onboarding`, `/invites/[token]`) — Story 7.7+ se necessário; MVP foca em `(app)/` que cobre 90%+ dos erros
- **Configurar Sentry alerts/notifications** — DevOps task, não Story de código
- **Source maps upload** para Sentry resolver minified stacktraces — já configurado pelo `@sentry/nextjs` automático em build

## Riscos

- **R1**: `Sentry.captureException` pode lançar erro próprio em ambientes sem DSN configurado. Mitigação: SDK já tem `enabled: false` quando DSN ausente, retorna silenciosamente.
- **R2**: Sentry quota free tier (5k erros/mês). Mitigação: validação manual reproduzirá UM erro; produção real ainda em pre-launch sem volume.
- **R3**: PII vazando no stacktrace. Mitigação: SDK Sentry padrão escrubba tokens, cookies, emails em `beforeSend`. ADR-021 documenta política.
