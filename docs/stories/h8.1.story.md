# Story h8.1: Vitest infra + cobertura ≥70% em lib financeira

## Status

Draft

## Story

**Como** mantenedor do KEYRA preocupado com regressão silenciosa em código financeiro,
**Eu quero** Vitest configurado + suíte de testes cobrindo `lib/money.ts`, validators Zod e Server Actions financeiras com ≥70% de cobertura,
**Para que** uma mudança em fórmula de fee/comissão/Decimal não passe despercebida pelo CI antes de chegar em produção e quebrar a confiança financeira da Camila.

## Complexidade

**T-shirt:** M+ (~6 pontos · 1.5 dia)

## Acceptance Criteria

### AC1 — Infra Vitest

1. `vitest.config.ts` na raiz de `apps/web/` com `@vitejs/plugin-react`, `globals: true`, `environment: 'node'` (default; tests específicos usam `'jsdom'` via comentário pragma).
2. `apps/web/package.json` ganha scripts: `test` (`vitest run`), `test:watch` (`vitest`), `test:coverage` (`vitest run --coverage`).
3. Path alias `@/*` funcionando dentro do Vitest (espelho do tsconfig).
4. CI workflow `.github/workflows/rls-tests.yml` ganha job `vitest` ANTES do RLS (paraleliza com `lint`/`typecheck`). Falha do Vitest bloqueia merge.
5. `pnpm test` na raiz delegando para `@keyra/web` via filter.

### AC2 — Cobertura mínima `lib/money.ts`

1. `formatBRL(decimal)`: 4 casos (zero, positivo, negativo, decimal grande > 99999.99).
2. `formatCentsBRL(cents)`: 4 casos (zero, positivo, negativo, > 9_999_999_99 cents).
3. `sumDecimal(arr)`: 3 casos (lista vazia, soma simples, soma com negativos).
4. Validar `ROUND_HALF_EVEN` explicitamente: `Decimal(2.5).toDecimalPlaces(0)` deve retornar 2 (não 3); `Decimal(3.5)` deve retornar 4.

### AC3 — Cobertura mínima `lib/validators/`

1. `appointment.ts`: schema válido + 3 inválidos (`startsAt > endsAt`, `professionalId` UUID malformed, `customerId` ausente).
2. `command.ts`: `addCommandItem` aceita item válido + rejeita `quantity <= 0` e `unitPrice < 0`.
3. `payment.ts`: `registerPayment` rejeita `gross < net + fee` (regra de consistência financeira).
4. `expense.ts`: rejeita `amount = 0` e `categoryId` ausente.

### AC4 — Cobertura mínima Server Actions financeiras

1. `createAppointment`: mock do supabase client, valida que snapshot de `price_snapshot`/`commission_snapshot` é tirado do `services` no momento da criação (não da referência).
2. `registerPayment`: validar cálculo de `fee` via `payment_methods.fee_percent_rate + fee_fixed_cents` com Decimal.js — golden test com 5 cenários canônicos.
3. `addCommandItem`: validar que `unit_cost` e `commission_rate` são copiados de `services` no INSERT.
4. `applyCommandDiscount`: rejeita desconto > subtotal (regra de invariante).

### AC5 — Coverage report e gate

1. `vitest --coverage` gera `coverage/lcov.info` e `coverage/index.html`.
2. Threshold mínimo: `lib/money.ts` ≥ 90%, `lib/validators/*.ts` ≥ 80%, Server Actions `**/actions.ts` ≥ 70%.
3. CI falha se threshold global < 70% ou se qualquer arquivo crítico cair abaixo de seu mínimo individual.
4. Relatório de cobertura publicado como artifact do GitHub Actions.

### AC6 — Documentação

1. `docs/testing/vitest-conventions.md` documenta padrão (estrutura `__tests__/`, naming `*.test.ts`, mocks).
2. README.md raiz atualizado com badge de coverage (Codecov free OU shield estático ler do CI).
3. Story de PR-template adicionada ao `.github/PULL_REQUEST_TEMPLATE.md`: checkbox "Coverage não regrediu".

## Dev Notes

### Decisões antecipadas

1. **Vitest sobre Jest:** ESM nativo + 2-3x mais rápido + integra direto com Vite (que está embaixo do Next 16 dev server). Jest precisa setup de Babel/ts-jest que adiciona ~1MB de config.
2. **Mocks de Supabase:** helper `__tests__/helpers/supabase-mock.ts` retorna chainable mock com `from().select().eq().single()` etc. Não usar MSW para esta camada — overhead desnecessário em testes unitários de Server Action.
3. **Não testar UI ainda:** esta story foca em lógica financeira. Testes de componente React (com `@testing-library/react`) ficam para story futura `h9.x` ou se algum AC pedir.
4. **Coverage threshold por arquivo:** Vitest 1.x suporta `thresholds` granulares por glob — usar para enforcing onde dói (money/validators/actions financeiras), não enforcing tudo.
