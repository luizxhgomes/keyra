# Story h8.2: Migração Vercel Pro + rotação de credentials

## Status

Draft

## Story

**Como** dono do KEYRA preparando go-live comercial,
**Eu quero** o projeto migrado para Vercel Pro e todas as credenciais expostas em chat rotacionadas,
**Para que** o produto possa aceitar pagamento sem violar a license do plano Hobby (uso pessoal apenas) e nenhum token comprometido permaneça válido.

## Complexidade

**T-shirt:** S (~3 pontos · 0.5 dia)

## Acceptance Criteria

### AC1 — Upgrade Vercel

1. Plano do projeto `keyra` migrado para **Pro ($20/mês)**.
2. Limites previstos: 1TB bandwidth/mês, log drains habilitados, deployment protection opcional.
3. Comprovante de upgrade salvo em `.keyra-secrets/vercel-pro-receipt.txt` (gitignored).

### AC2 — Rotação Vercel token

1. Token antigo de Vercel CLI revogado em [vercel.com/account/tokens](https://vercel.com/account/tokens).
2. Token novo gerado, salvo em `.keyra-secrets/vercel-token.txt` (chmod 600).
3. `scripts/sync-env.sh` continua funcionando.
4. Validação: `vercel whoami --token=$VERCEL_TOKEN` retorna `luizhenriquexpro`.

### AC3 — Rotação Supabase service_role + secret_key

1. `service_role` antiga revogada via Supabase Dashboard → Settings → API → "Reset service_role".
2. Nova `service_role` provisionada em `.keyra-secrets/supabase-service-role.key`.
3. Provisionar nova key nos 3 targets do Vercel (Production/Preview/Development) via `vercel env add SUPABASE_SERVICE_ROLE_KEY`.
4. Re-deploy de produção após rotação.
5. Validar que CI continua passando (RLS tests + lint + typecheck) com nova credencial.

### AC4 — Rotação Sentry token

1. Token antigo `sntrys_*` da Vercel revogado em [sentry.io/settings/account/api/auth-tokens/](https://sentry.io/settings/account/api/auth-tokens/).
2. Novo token com scopes mínimos: `project:releases` + `org:read`.
3. Re-provisionar em Vercel envs (Production apenas — source maps subindo).
4. Validar próximo build de produção sobe source maps (Sentry release com debug IDs visíveis).

### AC5 — Audit de credentials que ficaram

1. `scripts/check-tokens.sh` valida cada token contra sua API (Vercel `whoami`, Supabase `select 1`, Sentry `releases list`, Resend `domains list`).
2. Documento `docs/operations/credentials-audit-2026-05.md` com timestamp da rotação + escopo de cada token + última validação.
3. Lição salva: nunca compartilhar credentials em chat (já há memory `feedback_validate_creds_before_provisioning.md`).

## Dev Notes

### Cuidados na rotação

1. **Janela de inconsistência:** entre revogar credencial antiga e Vercel re-deployar com nova, há ~5min de Server Actions falhando com 401 do Supabase. Fazer rotação em janela de baixo tráfego (idealmente final de noite).
2. **Não rotacionar tudo simultaneamente:** rotacionar 1 por vez (Vercel → Supabase → Sentry) com smoke test de 2min entre cada. Se algo quebrar, a culpa é localizada.
3. **Resend está OK:** chave atual `re_9UB...` foi gerada em 2026-04-30 limpa, sem comprometimento conhecido. Conferir scopes mas não revogar prematuramente.

### Comprovação anti-vazamento

Rotação inicia o relógio: se chave antiga aparecer em commit/log/issue depois de hoje, é vazamento novo (não residual). Documentar timestamp da rotação no audit.
