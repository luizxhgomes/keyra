# Story h8.3: Vercel Cron healthcheck (anti-pause Supabase Free)

## Status

Draft

## Story

**Como** operador do KEYRA usando Supabase Free,
**Eu quero** um cron Vercel que pinga o banco a cada 6 horas,
**Para que** a regra de pause após 7 dias sem requests nunca seja atingida e o produto continue acessível para a idealizadora e clínicas-piloto sem intervenção manual.

## Complexidade

**T-shirt:** XS (~1 ponto · 30 min)

## Acceptance Criteria

### AC1 — Endpoint healthcheck

1. `app/api/healthcheck/route.ts` implementado como Route Handler GET.
2. Retorna JSON `{ status: 'ok', supabaseLatencyMs: number, timestamp: ISO8601 }`.
3. Faz 1 query trivial via `createAdminClient` (bypass RLS, server-only): `SELECT count(*) FROM organizations LIMIT 1`.
4. Latência da query medida e retornada.
5. Em caso de erro (Supabase down/pause prestes a acontecer): retorna 503 com `{ status: 'degraded', error: string }` — mas SEM logar credenciais ou stacktrace sensível.

### AC2 — Vercel Cron config

1. `vercel.json` (ou `apps/web/vercel.json`) ganha entrada:
   ```json
   {
     "crons": [
       { "path": "/api/healthcheck", "schedule": "0 */6 * * *" }
     ]
   }
   ```
2. Schedule: a cada 6 horas (4 hits/dia × 7 dias = 28 hits/semana, longe do limite Hobby/Pro).
3. Cron protegido contra acesso público: header `x-vercel-cron-signature` validado no Route Handler (Vercel envia automaticamente).
4. Acesso direto sem signature retorna 401.

### AC3 — Observabilidade

1. Sentry breadcrumb `cron.healthcheck.executed` com latência e timestamp.
2. Se latência > 1000ms, capture warning (Supabase devagar — sinal de pause iminente ou degradação).
3. Próxima sessão de revisão de prod inclui leitura desses breadcrumbs.

### AC4 — Documentação

1. `docs/operations/cron-jobs.md` lista todos os crons ativos com responsabilidade.
2. Entrada nesse doc para `healthcheck`: propósito, schedule, métrica esperada.

## Dev Notes

### Por que `count(*)` e não `select 1`

Supabase considera "request real" qualquer query SQL — `select 1` no Postgres direto via PostgREST seria barato mas sem garantia de que o auth schema tá responsivo. Query em tabela real (com RLS bypass via admin client) confirma o caminho completo de produção.

### Por que 6h e não 24h

Margem de segurança: limite Free é 7 dias. Com 6h de intervalo, mesmo que falhem 4 hits seguidos, ainda há 5+ dias antes do pause. Hits adicionais não são caros — Vercel Pro permite ~unlimited cron.
