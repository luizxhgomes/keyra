# Story h8.4: LGPD toolkit (consent + export + anonymize + DPO + terms)

## Status

Draft

## Story

**Como** controlador de dados pessoais (Camila e seus pacientes),
**Eu quero** o sistema implementar os 5 direitos da LGPD (consentimento, acesso, esquecimento, DPO, transparência),
**Para que** o KEYRA possa ser oferecido publicamente sem violar Lei 13.709/2018 e a Camila possa demonstrar conformidade se for fiscalizada.

## Complexidade

**T-shirt:** L (~8 pontos · 2 dias)

## Acceptance Criteria

### AC1 — Consent records (registro de aceite)

1. Migration nova: tabela `consent_records (id uuid PK, org_id uuid NOT NULL FK, user_id uuid NOT NULL FK auth.users, terms_version text NOT NULL, accepted_at timestamptz NOT NULL DEFAULT now(), ip_address inet NOT NULL, user_agent text NOT NULL)`.
2. RLS: cada user vê só seus próprios consent_records; org owner vê todos da org.
3. Server Action `recordConsent(termsVersion: string)` chamada no signup e a cada mudança de termos.
4. `requireAuth()` redireciona para `/legal/aceitar-termos` se a versão atual não bate com o último `consent_records` do user.

### AC2 — Direito de acesso (export user data)

1. Página `/configuracoes/meus-dados` com botão "Exportar todos os meus dados (LGPD Art. 18)".
2. Server Action `exportUserData(userId)` retorna ZIP com:
   - `profile.json` (auth.users info — sem hash de senha)
   - `memberships.csv` (orgs onde é membro)
   - `customers.csv` (se for owner: pacientes da org; se professional: vazio ou só os atendidos por ele)
   - `appointments.csv`, `commands.csv`, `payments.csv`, `transactions.csv` filtrados por user/org
   - `consent_records.csv` (histórico de aceite)
   - `audit_log.csv` (entries onde `user_id = currentUser`)
3. Export gerado em background via Vercel function (timeout até 60s; se exceder, usa Inngest — mas Inngest é Phase 5, então MVP usa sync com paginação).
4. Toast "Seus dados foram baixados — guarde com cuidado, contém informações pessoais."
5. Audit_log registra `action='data_export'`.

### AC3 — Direito ao esquecimento (anonymize customer)

1. Em `/pacientes/[id]`, botão **destrutivo** "Excluir dados pessoais (LGPD Art. 18)".
2. AlertDialog confirma: "Os dados pessoais (nome, telefone, CPF, email, endereço) serão substituídos por hash determinístico. **Estatísticas (atendimentos, valores) serão preservadas**. Esta ação NÃO PODE SER DESFEITA."
3. Server Action `anonymizeCustomer(customerId)`:
   - Atualiza `customers` setando `name='[REDACTED-{hash8}]'`, `phone=NULL`, `email=NULL`, `cpf_encrypted=NULL`, `address=NULL`, `anonymized_at=now()`, `anonymized_by=currentUser`.
   - Hash determinístico de 8 chars baseado em `id` (não em PII original — para preservar idempotência).
   - Audit_log captura entry com `before_data` (SEM PII — só campos não-sensíveis) e `after_data` (com hash).
   - Não delete linhas relacionadas em `appointments`, `commands`, etc. (preserva agregados financeiros e histórico).
4. Customer anonimizado fica filtrável mas sem nome/contato — cliente "fantasma" no histórico.
5. Lista de pacientes mostra "[REDACTED-abc12345]" em vez de nome.

### AC4 — DPO contact + Política de Privacidade + Termos de Uso

1. Página `/legal/dpo` com email do DPO (configurável via env `DPO_EMAIL`, default `dpo@usekeyra.com`).
2. Página `/legal/privacidade` com Política de Privacidade em pt-BR (template revisto por advogado).
3. Página `/legal/termos` com Termos de Uso.
4. Footer global com links para essas 3 páginas (mesmo nas rotas autenticadas — Sidebar tem item "Legal" no rodapé do menu).
5. Migration nova: tabela `terms_versions (id text PK, content_md text, published_at timestamptz, kind text CHECK kind IN ('privacy', 'terms'))`.
6. Seed inicial com versão `2026-05-02-v1`.

### AC5 — Re-aceite de termos ao mudar versão

1. Server Action `getCurrentTermsVersion(kind)` retorna versão mais recente.
2. Layout `(app)/layout.tsx` checa: se `consent_records` mais recente do user tem `terms_version` antiga vs current → redirect para `/legal/aceitar-termos?next=...`.
3. Página `/legal/aceitar-termos` mostra diff dos termos novos + botão "Aceitar e continuar".
4. Aceitar grava novo `consent_records` e libera redirect.

### AC6 — Audit + observabilidade

1. Audit_log captura `action='data_export'`, `action='customer_anonymized'`, `action='consent_recorded'`.
2. Sentry breadcrumb em cada uma dessas ações (sem PII).
3. Métricas no PostHog (se já implementado por h8.5): `lgpd_export_requested`, `lgpd_anonymize_executed`.

## Dev Notes

### Hash determinístico

```typescript
// lib/lgpd/anonymize.ts
import crypto from 'crypto';

export function deterministicHash(id: string): string {
  return crypto.createHash('sha256').update(id).digest('hex').slice(0, 8);
}
```

Hash baseado em `customer.id` (não em CPF/nome) — garante idempotência (segundo anonymize do mesmo customer dá mesmo hash) e não vaza informação reversível.

### Estatísticas preservadas

Após anonymize, o cliente vira `[REDACTED-abc12345]` mas todas as métricas continuam:
- DRE inclui as transações (não são pessoais)
- Lucro por serviço continua exato
- Histórico de comandas existe (sem nome do cliente)
- Agendamentos passados visíveis (filtro mostra "[REDACTED-abc12345]")

Cliente novo com dados não-sensíveis pode ser criado se ela voltar — o registro original fica como ghost histórico.
