# Story h8.5: PostHog instrumentation + 5 eventos canônicos (NorthStar TTFL mensurável)

## Status

Draft

## Story

**Como** product owner do KEYRA querendo validar a tese central,
**Eu quero** PostHog Cloud instrumentado capturando 5 eventos canônicos do funil de ativação,
**Para que** o NorthStar TTFL (Time To First Lucro) seja mensurável objetivamente e decisões pós-piloto sejam baseadas em dado, não em achismo.

## Complexidade

**T-shirt:** M (~5 pontos · 1 dia)

## Acceptance Criteria

### AC1 — PostHog SDK instalado

1. `posthog-js` instalado em `apps/web/package.json`.
2. `lib/analytics/posthog.ts` exporta `analytics.capture(event, props)` com fallback no-op se `NEXT_PUBLIC_POSTHOG_KEY` ausente.
3. PostHog Provider envolvendo `(app)/layout.tsx` (não em `(auth)` ou `onboarding/` — opt-in só após login).
4. Identificação automática: `analytics.identify(user.id, { org_id, role })` no boot do AppShell.
5. Auto-capture desabilitado (privacidade > telemetria automática). Só eventos explícitos.

### AC2 — Os 5 eventos canônicos

| # | Evento | Quando dispara | Props |
|---|--------|----------------|-------|
| 1 | `signup` | Server Action `signInWithOtpAction` quando user é criado pela 1ª vez | `{ source: 'magic_link' \| 'invite' }` |
| 2 | `onboarding_complete` | Server Action `createOrgWithOwner` retorna sucesso | `{ org_id, time_from_signup_minutes }` |
| 3 | `first_appointment_done` | Server Action `changeAppointmentStatus(_, 'done', _)` quando é o 1º `done` daquela org | `{ org_id, time_from_onboarding_hours }` |
| 4 | `first_payment_recorded` | Server Action `registerPayment` quando é o 1º payment daquela org | `{ org_id, payment_method, gross_cents }` |
| 5 | `first_dashboard_with_lucro` | Boot do `/dashboard` quando `KPI lucro > 0` E é a 1ª vez que essa condição é verdade naquela org | `{ org_id, time_from_signup_hours, lucro_cents }` |

### AC3 — Detecção de "primeira vez"

1. Coluna `organizations.posthog_milestones jsonb DEFAULT '{}'::jsonb` armazena os 5 marcos com timestamp de cada.
2. Server Action lê o JSONB; se evento ainda não disparou, dispara + UPDATE + retorna.
3. Pattern compatible com concorrência: usa `UPDATE ... WHERE posthog_milestones->>{evento} IS NULL RETURNING` para race-safety.

### AC4 — TTFL dashboard interno (`@aiox-master` only)

1. Página `/admin/funnel` (gated por `role='superadmin'` — coluna nova em `memberships`) com:
   - Tabela: org_id, signup_at, onboarding_at, first_done_at, first_payment_at, first_lucro_at, ttfl_hours
   - Métrica agregada: TTFL médio + p50 + p90 das últimas 30 orgs
   - Benchmark: PRD pede TTFL < 24h
2. Acesso somente para o owner do KEYRA inicialmente; LGPD: dados agregados, sem PII de pacientes.

### AC5 — Privacy & opt-out

1. Página `/configuracoes/privacidade` com toggle "Compartilhar dados de uso anonimizados para melhorar o KEYRA" (default ON, mas conscientizado no aceite de termos).
2. Toggle OFF → `analytics.optOut()` + nenhum evento futuro disparado.
3. Eventos NUNCA carregam PII de pacientes (nome, CPF, telefone). Só agregados financeiros e UUIDs internos.

### AC6 — Documentação

1. `docs/operations/analytics-events.md` lista cada evento, propriedades, quando dispara, dashboard onde aparece.
2. Convention: nomes `snake_case`, props com sufixo de unidade (`_cents`, `_hours`, `_minutes`).

## Dev Notes

### PostHog Cloud free tier

- 1M events/mês free.
- 5 eventos × 100 orgs × 30 dias = ~15K eventos/mês inicialmente. Margem de 60x antes de pagar.
- Cloud → DPA com PostHog (data processing agreement) — `@compliance-br` confere se está alinhado com LGPD (PostHog é EU-hosted, OK para LGPD).

### Por que JSONB e não tabela

Coluna `posthog_milestones` em `organizations` é mais simples que tabela `posthog_events`: race-safety via SQL atomic + 1 query a menos. A "tabela completa" de eventos vive no PostHog mesmo — não é responsabilidade do nosso DB.

### Caso edge: org com membros migrando

Se Camila convida nova esteticista após primeiro_lucro_at, eventos não re-disparam (já marcados na org). Comportamento correto — TTFL é por org, não por user.
