# Story h8.8: DR runbook + RPO/RTO testados

## Status

Draft

## Story

**Como** operador do KEYRA preparando go-live comercial,
**Eu quero** um runbook de Disaster Recovery testado em ambiente real,
**Para que** RPO ≤ 24h e RTO ≤ 4h (NFR-AV-02/03 do PRD) sejam comprovados, não promessas.

## Complexidade

**T-shirt:** S (~3 pontos · 0.5 dia)

## Acceptance Criteria

### AC1 — Runbook documentado

1. `docs/operations/disaster-recovery.md` com:
   - **RPO target:** ≤ 24h (Supabase Pro inclui daily backup) → ≤ 1h se PITR habilitado
   - **RTO target:** ≤ 4h
   - **Cenários cobertos:**
     - C1: Vercel down (degradação parcial — landing fora, app fora — só estático no Cloudflare)
     - C2: Supabase down (app sobe mas reads/writes falham — modo degradado com mensagem amigável)
     - C3: Sentry down (não-crítico — só perde observabilidade)
     - C4: DB corrompido / migration ruim em prod (pior caso — restore via PITR)
     - C5: Vazamento de dados / breach de credentials (rotação emergencial)
   - **Para cada cenário:** detection → containment → restoration → comunicação à idealizadora → post-mortem template

### AC2 — Backup test

1. Confirmar que Supabase Free faz backup diário (Settings → Database → Backups).
2. Realizar restore de teste em projeto sandbox: criar projeto temporário Supabase, restaurar último backup do `keyra-br`, validar que pelo menos 5 tabelas core (organizations, memberships, customers, services, transactions) chegam intactas.
3. Documentar tempo gasto (esperado < 4h target).

### AC3 — Migração para Supabase Pro

1. Avaliar custo: $25/mês = 1 PITR de até 7 dias = RPO ≤ 1h.
2. Decisão registrada (`@aiox-master`): manter Free até primeiro paying customer; upgrade para Pro junto com Story `h8.2` se desejado.
3. Se Free: RPO = 24h aceito + documentado para clientes piloto.

### AC4 — Comunicação à idealizadora

1. Template de mensagem WhatsApp para enviar à idealizadora caso de outage:
   ```
   Olá! O KEYRA está com instabilidade desde {hora}. Estimativa: {ETA}.
   Sua agenda do dia continua salva — assim que voltar, está intacta.
   Te aviso aqui quando normalizar.
   ```
2. Template de post-mortem (markdown) para outages que durem > 30 min.

### AC5 — Validação semestral

1. Adicionar lembrete em `docs/operations/calendar.md` (ou cron Vercel anual) para repetir teste de restore a cada 6 meses.
2. Audit: cada teste registra timestamp + duração no runbook.

## Dev Notes

### Por que importa para go-live

Sem runbook, primeira outage = improviso = downtime maior + comunicação ruim com a idealizadora = perda de confiança. Runbook é seguro contra panic.

### Cenário mais provável

C2 (Supabase down) é o mais provável e o mais impactante. Mitigação: app deve renderizar mensagem amigável (`<ErrorMessage>`) em vez de crashar — Story já existente `7.4` cobriu `not-found.tsx` + `global-error.tsx`. **Validar que esse boundary captura também erro de Supabase em Server Action.**
